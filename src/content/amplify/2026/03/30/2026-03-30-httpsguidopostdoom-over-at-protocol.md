<p>Someone ran Doom through DNS. <a href="https://blog.rice.is/post/doom-over-dns/">Adam Rice stored the entire game in TXT records</a> and it actually worked. I saw that and - being at an AT Protocol conference - thought: can the AT Protocol do the same thing? Store the game and actually play it, with every player input and every rendered frame traveling through the protocol?</p>
<p>The answer is yes. Sorta. It plays terribly. No-one should want this. But it works. <a href="https://doom.singi.dev/">You can try it yourself.</a></p>
<p>Fair warning: this is a technical post. If you’re here for the details, you’re in the right place.</p>
<h2>The data flow</h2>
<p>Every piece of data travels through AT Protocol records. The architecture is a hybrid: Jetstream for input delivery (because the player’s PDS is on bsky.social), direct localhost PDS polling for frame delivery (because it’s faster than waiting for the relay network to crawl a new self-hosted PDS).</p>
<figure><img alt="Data flow diagram: Browser sends keyboard input via WebSocket to the client server, which writes input records to the player's PDS on bsky.social. Jetstream delivers those events to the game server, which runs the Doom WASM engine, encodes the frame as PNG, and writes it to the game server's self-hosted PDS. The client server polls the localhost PDS for new frame records and sends the PNG blob via WebSocket to the browser's canvas." src="https://gui.do/_astro/diagram-dataflow.D3IaoEDp_2dqSjM.webp?dpl=69c9f77bbeb0360008d891b4"/><figcaption>Data flow diagram: Browser sends keyboard input via WebSocket to the client server, which writes input records to the player's PDS on bsky.social. <p>When you press a key:</p>
<ol> <li>Your browser captures the keyboard event and encodes it as a bitmask</li> <li>The client server writes that bitmask as a <code>dev.singi.doom.input</code> record to your PDS (probably bsky.social)</li> <li>Jetstream (AT Protocol’s real-time event stream) delivers that record to the game server</li> <li>The game server queues your input and flushes it before the next tick batch</li> <li>The Doom WASM engine ticks (5 ticks per batch), renders a frame at 320x200</li> <li>A custom PNG encoder compresses the frame to 8-37KB</li> <li>The game server writes that PNG as a blob-backed <code>dev.singi.doom.frame</code> record to its own self-hosted PDS</li> <li>The client server polls the localhost PDS every 100ms for new frame records</li> <li>It fetches the PNG blob and sends it to your browser via WebSocket</li> <li>Your browser renders it to a canvas</li> </ol>
<p>Every step goes through AT Protocol records. That’s the whole point (and the whole self-induced problem).</p>
<h2>The game engine</h2>
<p>I’m using <a href="https://github.com/ozkl/doomgeneric">doomgeneric</a>, a portable C implementation of the Doom engine designed for “bring your own I/O.” It exposes six callbacks that you implement yourself: init, draw frame, sleep, get ticks, get key, and set window title. Most of those are no-ops here because there’s no actual display to manage.</p>
<p>Compiled to WebAssembly via Emscripten, the engine produces a 325KB <code>.wasm</code> file running at 320x200 resolution.</p>
<h3>Why it runs in a Worker thread</h3>
<p>The WASM <code>doom_tick()</code> call is synchronous and takes 2-20ms depending on the scene complexity. That doesn’t sound like much, but at the frame rates I’m targeting it would block Node.js’s event loop enough to starve HTTP request handling. Running the engine in a Worker thread keeps the main thread responsive for WebSocket connections, OAuth callbacks, and frame delivery.</p>
<p>Key events from Jetstream arrive asynchronously while the engine may be mid-tick. Sending key events to the Worker during an active <code>doom_tick()</code> caused WASM memory corruption (“memory access out of bounds”). The fix: queue key events in the main thread and flush them to the Worker only between tick batches.</p>
<h3>Frame encoding</h3>
<p>Doom renders with a 256-color palette, which is great for indexed-color PNG. Claude Code wrote a minimal PNG encoder that:</p>
<ol> <li>Scans the RGBA framebuffer for unique colors (always under 256 for Doom)</li> <li>Builds a palette</li> <li>Writes palette-indexed scanlines</li> <li>Compresses with zlib deflate (level 6)</li> </ol>
<p>Result: 8-37KB per frame, compared to ~90KB with standard RGB PNG. Encoding takes about 5ms. The frames fit comfortably within AT Protocol blob size limits.</p>
<p>The browser captures keyboard events and encodes them as a 19-bit bitmask:</p>
<figure><img alt="Bitmask layout diagram: bits 0-3 for movement (forward, backward, left, right), bits 4-7 for actions (fire, use, strafe, speed), bits 8-15 for weapon selection (8 weapon slots), and bits 16-18 for menu (escape, enter, tab)." src="https://gui.do/_astro/diagram-bitmask.DA5JCvBN_1sDuJi.webp?dpl=69c9f77bbeb0360008d891b4"/><figcaption>Bitmask layout diagram: bits 0-3 for movement (forward, backward, left, right), bits 4-7 for actions (fire, use, strafe, speed), bits 8-15 for weapon selection (8 weapon slots), and bits 16-18 for menu (escape, enter, tab).</figcaption></figure>

<p>Key state changes (not every keypress) are batched and written to the player’s PDS as <code>dev.singi.doom.input</code> records every 500ms. That keeps the write rate at ~2 records/sec.</p>
<p>The game server subscribes to those input records via Jetstream, diffs each bitmask against the previous state to generate press/release events, and queues them for the next tick batch.</p>
<h3>Stale event filtering</h3>
<p>Jetstream replays events from its buffer on reconnection. Without filtering, old input records from previous sessions would replay, navigating through menus and starting games automatically. Both the server (inputs) and client (frames) skip any event with a <code>createdAt</code> timestamp older than 10 seconds.</p>
<h2>Authentication</h2>
<p>Players authenticate via AT Protocol OAuth with granular scopes. The consent screen only asks for permission to write <code>dev.singi.doom.input</code> records. No access to your posts, profile, or DMs. If a PDS doesn’t support granular scopes, the client falls back to <code>transition:generic</code>.</p>
<p>The OAuth keypair is persisted to disk so it survives container restarts. Without persistence, each restart generates a new key, but the player’s PDS caches the old JWKS, causing “invalid_client” errors until the cache expires.</p>
<p>The game server itself uses an app password on its self-hosted PDS, so no OAuth needed for server-to-PDS writes on localhost.</p>
<h2>Why a self-hosted PDS</h2>
<p>Early versions wrote frame records to bsky.social, which has rate limits: 5,000 write points per hour, with each record creation costing 3 points. Quick math: at just 2 frames per second, that’s 6 points per second, 360 per minute, 21,600 per hour. You blow through the hourly limit in about 14 minutes.</p>
<p>And on top of the documented limits, there’s undocumented anti-abuse throttling. One test session wrote 1,664 records in 2 minutes and locked the account for over 12 hours, well beyond the documented rate limit windows.</p>
<p>I now also cycle through multiple game bot accounts, automatically switching when one approached its rate limit budget. Only needed if some wacko plays this for over 45 minutes so I think we’re ok.</p>
<p>This runs on a dedicated Hetzner CX23 VPS running a self-hosted AT Protocol PDS alongside the game engine. Frame writes go to <code>localhost:3000</code> with near-zero latency and no rate limits. The PDS federates with the AT Protocol network via <code>plc.directory</code> for identity resolution.</p>
<p>This is arguably the architecturally correct setup anyway. The game server is the authority on frame data, so it should host that data on its own PDS. Federation at work!</p>
<h2>Jetstream and the hybrid architecture</h2>
<p>The first implementation used <code>listRecords</code> polling every 200ms for everything. Three problems:</p>
<ol> <li><strong>Record ordering was wrong.</strong> TIDs (AT Protocol’s timestamp-based IDs) sort differently than I expected, so records arrived out of order.</li> <li><strong>Wasted API calls.</strong> Polling burns requests even when nothing changed.</li> <li><strong>Latency floor.</strong> 200ms minimum, no matter what.</li> </ol>
<p>Switching to Jetstream for input delivery solved all three. It’s the AT Protocol’s real-time event firehose, delivered over WebSocket. Events arrive within ~50-200ms of record creation. The server subscribes filtered by the player’s DID and the <code>dev.singi.doom.input</code> collection.</p>
<p>But: frame delivery went <em>back</em> to polling. A newly self-hosted PDS needs to be crawled by the relay network before Jetstream sees its events, and that can take minutes or fail silently. Since the game server and client run on the same machine as the PDS, polling localhost at 100ms is actually faster than waiting for relay propagation.</p>
<p>So the final architecture is a pragmatic hybrid: Jetstream where it’s the only option (cross-PDS input delivery from bsky.social), direct PDS polling where it’s faster (same-machine frame delivery).</p>
<p>Jetstream might be one of the most underappreciated pieces of AT Protocol infrastructure. It’s a general-purpose real-time event system that works for any application built on AT Protocol records. It’s what got this from “completely unplayable” to “technically playable”. If you’re generous with the word “playable”…</p>
<h2>The lexicons</h2>
<p>Five custom lexicons define the game’s data schema:</p>
<table><tr><th>Lexicon</th><th>Purpose</th></tr><tbody><tr><td><code>dev.singi.doom.defs</code></td><td>Shared types: key bitmask, frame metadata, encoding tokens</td></tr><tr><td><code>dev.singi.doom.session</code></td><td>Game session metadata: WAD, player DID, status, tick rate</td></tr><tr><td><code>dev.singi.doom.input</code></td><td>Player input: array of key bitmasks per record</td></tr><tr><td><code>dev.singi.doom.frame</code></td><td>Rendered frame: PNG blob reference, dimensions, sequence number</td></tr><tr><td><code>dev.singi.doom.artifact</code></td><td>Game asset storage for chunked blobs (the “DNS mode”)</td></tr></tbody></table>
<p>The <code>artifact</code> lexicon enables a second mode inspired directly by the DNS project: store the entire Doom WAD and engine as chunked blobs on a PDS, fetch them, and run locally. The AT Protocol equivalent of Adam Rice’s 2,000 DNS TXT records, except you only need about 5 blob records.</p>

<h3>What would it take for higher framerates?</h3>
<p>Not much you can do within the protocol constraints.</p>
<p>The frame delivery side is already fast (localhost polling at 100ms). The remaining bottleneck is Jetstream input delivery (~50-200ms), which you can’t eliminate because the player’s PDS is on some remote PDS (e.g. bsky.social).</p>
<p>Some things that could help:</p>
<ul> <li><strong>Colocating with a Jetstream endpoint</strong> could shave network latency off the input hop.</li> <li><strong>Running your own relay</strong> would remove the relay hop entirely, but at that point you’re basically building a direct connection with extra steps.</li> <li><strong>Predictive rendering on the client</strong> could mask latency by running the engine locally and reconciling with server frames when they arrive. Possible, but complicated.</li> </ul>
<p>The realistic ceiling is probably ~15-20fps with favorable Jetstream latency. For comparison, a direct WebSocket between client and server would give 60fps easily. But doing that defeats the entire point of the project.</p>
<p><strong>AT Protocol is very flexible.</strong> Custom lexicons, blob storage, OAuth with granular scopes, federation… the building blocks for applications far beyond social media are already there. I’m building <a href="https://barazo.com">Barazo</a> and <a href="https://sifa.id">Sifa ID</a> as serious projects on the protocol. This was the fun side project that showed me how much room there is to experiment.</p>
<p><strong>Self-host your PDS if you’re building anything non-trivial.</strong> The AT Protocol is designed for federation. Running your own PDS isn’t just about avoiding rate limits, it’s about being the authority for your data. The game server produces frame data, so it should host that data on its own PDS. This also eliminates dependency on relay crawling delays.</p>
<p><strong>Jetstream deserves more attention.</strong> Any application built on AT Protocol records can use it for real-time event delivery. Feed generation is the obvious use case, but it works for anything. And it’s what makes this game “playable”.</p>
<p><strong>Rate limits are the real boss.</strong> The hardest part wasn’t the WASM engine or the protocol integration. It was staying within bsky.social’s rate limits for both the server and the user PDS. The documented limits (5,000 points/hour) are only part of the story: there’s also undocumented anti-abuse throttling that can lock accounts after burst writes.</p>
<p><strong>WASM engines are surprisingly fragile at high tick rates.</strong> The doomgeneric WASM engine crashes with memory access violations when key events are sent mid-tick or when tick rates exceed what the engine expects. Careful sequencing (queue events, flush before tick, await completion) is the only thing that works reliably.</p>
<p>Also: DO NOT use your main account/PDS for testing. I got rate-limited for 12 hours and couldn’t use my account during #ATmosphereConf. Not ideal.</p>
