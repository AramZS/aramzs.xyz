<h1>Introducing Hubble: A Public Mirror for the Whole Atmosphere</h1>
<p><em>This is a guest post from <a href="https://bsky.app/profile/did:plc:hdhoaan3xa3jiuq4fg4mefid">fig</a>, creator and maintainer of the microcosm infrastructure for building great atproto apps, <a href="https://updates.microcosm.blue/3mhj675s55c2a">originally published on the microcosm blog</a>. We're thrilled to be supporting fig's work building a more resilient ecosystem with Hubble.</em></p>
<p>Every account in the Atmosphere lives in a Personal Data Server (PDS) somewhere. Whether you run your own (like <a href="https://bsky.app/profile/doctorow.pluralistic.net/post/3mgdl7pj3z22q">Cory Doctorow</a>), team up with friends to share a server, or park your account at a PDS from Bluesky, <a href="https://blacksky.community/">Blacksky</a>, <a href="https://northskysocial.com/">Northsky</a>, <a href="https://www.eurosky.tech/">Eurosky</a>, <a href="https://npmx.dev/">npmx.social</a>, <a href="https://selfhosted.social/">selfhosted.social</a>, <a href="https://pckt.blog">pckt.blog</a>, <a href="http://margin.at">margin.at</a>, <a href="http://sprk.so">sprk.so</a>, <a href="http://tngl.sh">tngl.sh</a>, <a href="https://teal.fm">teal</a><em>(deep breath)—</em> your app experience is the same. Your PDS logs you in, and holds your Bluesky posts, Leaflet blogs, and alllll your other public data for atproto apps.</p>
<p>But what happens when a PDS goes offline?</p>
<p>For you as a user, this is recoverable! …<em>if</em> you’ve taken some preparations. It's easier to move when your PDS is online, but with a <a href="https://www.da.vidbuchanan.co.uk/blog/adversarial-pds-migration.html#rotation-key">rotation key</a> and a backup of your data, you're in complete control. Tools like <a href="https://pdsmoover.com/">PDS MOOver</a>—a community-built multi-purpose account management tool—have made this preparation <em>much</em> easier than it used to be.</p>
<p>For developers building in the Atmosphere, the answer is less comfortable. An offline PDS is like a hole cut out of the database. If your app has saved a copy of the data it needs from it, you might be fine. But if you need anything from it, or spin up a new app and backfill from scratch: unless you can find a copy from somewhere, you are simply out of luck. Your picture of the network will be a bit different from everyone else's, potentially permanently.</p>
<p>It wasn’t always like this.</p>
<p>In the bad old days of early Atmosphere, <a href="https://atproto.com/guides/glossary#relay">relays</a> stored a local copy of every user’s data repository (“repo”). This was great for apps! Devs could backfill the entire network from a relay, no matter whether any PDS was online or not.</p>
<p>Unfortunately this setup was less great for the relays themselves, because keeping copies of <em>tens of millions</em> of repositories is <a href="https://whtwnd.com/bnewbold.net/3kwzl7tye6u2y">expensive</a> to store and run.</p>
<p>Last year the <a href="http://github.com/bluesky-social/proposals/tree/main/0006-sync-iteration">sync1.1</a> protocol update changed the firehose game: now you can fully authenticate repository data updates <em>without</em> keeping your own full copy for proof! Smart cryptography! All relays in the network now run this way, and are called <em>non-archival</em>: lightweight and <a href="https://bsky.app/profile/bad-example.com/post/3mfkrfvy3ok2u">cheap</a> to run, but only retaining recent events, not the full history of the network.</p>
<p>These days, atproto apps backfill by crawling the live network themselves, instead of replaying from an archival relay. Initial repository contents are downloaded direct-from-PDS, and Bluesky’s <a href="https://github.com/bluesky-social/indigo/tree/main/cmd/tap#readme">Tap</a> and other great tools have emerged which help developers manage the backfill process. But Tap can’t bring unreachable PDSes back from the dead offline.</p>
<p>When a PDS disappears, those repos are gone.</p>
<p><strong>Hubble</strong> is a new open-source project to build and operate a whole-Atmosphere public data mirror, synchronizing every atproto repository in real-time, keeping public data available even when a PDS goes down.</p>
<p>I've wanted to build this for a while! I build and run <a href="https://constellation.microcosm.blue/">various</a> <a href="https://spacedust.microcosm.blue/">full</a>-<a href="https://lightrail.microcosm.blue/">network</a> <a href="https://ufos-api.microcosm.blue/#GET/collections">indexes</a>, <a href="https://relay2.fire.hose.cam/">relays</a>, <a href="https://jetstream2.fr.hose.cam/">and</a> <a href="https://us-west.plc.wtf/">other</a> <a href="https://plc.wtf/">atproto</a> <a href="https://tangled.org/microcosm.blue/lightrail">infrastructure</a>, including an identity-and-record edge cache (called <a href="https://slingshot.microcosm.blue/#description/slingshot-edge-record-and-identity-cache">Slingshot</a>) which already <em>helps a bit</em> with PDS availability gaps—<em>if</em> it has already cached the data you want.</p>
<p>A full mirror of the network is a step up from that and a step-change in what it brings: account recovery for users, data availability for apps, and complete snapshots for researchers. But it’s a bigger, riskier project, with potentially high operating costs and no obvious direct revenue model. Hard to take the leap without some support.</p>
<p>That support came together as a $20,000 grant from Bluesky for Hubble’s development and one year of operation. The structure of the grant aligns incentives: operating costs come out of the total amount, so the more efficient I can make it, the better for me. (and if DRAM keeps rocketing in price, …uh oh). We set out some acceptance criteria for Hubble’s launch (and the final payout), and then I run it for 12 months.</p>
<p>What happens after 12 months? It’s all open-source, so anyone (including me!) will be able to keep a Hubble server running. And I plan to! But <em>if</em> after a year i <em>can't</em> continue operating it sustainably, Bluesky will pick up the live instance to keep it online. Hubble itself won’t go down like an offline PDS!</p>
<p>I think it’s a pretty neat funding arrangement! Substantial support without giving up independence, while keeping a collaborative view toward long-term stability for the ecosystem.</p>
<p>One thing to underscore: because all public data in atproto is cryptographically signed by the account it’s from, Hubble cannot tamper with it. This is the <em>Authenticated</em> in the Authenticated Transfer Protocol: copies of public data you get from anywhere are verifiable, even if you can’t reach the original PDS it came from.</p>
<p>Hubble will mirror every public data repository in the Atmosphere, like relays did before the non-archival switch. Repositories contain almost everything you might think of as “your data” that you share publicly: your profile, all your likes, posts, replies, follows, recipes, RSVPs, …</p>
<p>But it’s worth noting a few things that Hubble will <strong>not</strong> store:</p>
<ul> <li><strong>No blobs:</strong> Images, videos, and other media live separately from repos. It takes a lot of storage to keep 43+ million repositories, and blobs take <em>orders of magnitude</em> more space than that! (think: <em>petabytes</em>). Future work for Hubble.</li> <li><strong>No private app data</strong>: Things like mutes and bookmarks on Bluesky that aren’t part of your public data repository will not be stored by Hubble. (This isn’t just Hubble being virtuous, it literally <em>cannot access</em> any non-public data!)</li> </ul>
<p>Hubble will fully respect content deletion. Deleted records are deleted from the mirror, and deleting your Atmosphere account removes your entire data repository. Data from inactive accounts won't be accessible from unauthenticated public endpoints. <strong>Hubble mirrors the current state of the network, not its history.</strong></p>
<p>For personal account recovery, Hubble will launch with a public website where you can find and download your data: <a href="https://hubble.microcosm.blue">hubble.microcosm.blue</a>. The archive you’ll receive is <em>exactly</em> the same as what you get when exporting your data from the Bluesky app, for example. You can import this archive when moving to a new PDS to restore all your public data.</p>

<p>I’m fig (or Phil) (they/them), and I started <a href="https://microcosm.blue">microcosm.blue</a> to build and run community infrastructure for the Atmosphere. In the year since its launching the first service (<a href="https://constellation.microcosm.blue/">Constellation</a> universal backlinks), microcosm has grown to serve hundreds of requests per second from dozens of apps you might have heard of, like Blacksky, npmx.dev, and PDS MOOver.</p>
<p>Microcosm already hosts other independent protocol-level infrastructure—multiple relays, PLC mirrors—along with new full-network API services like <a href="https://slingshot.microcosm.blue">Slingshot</a>, <a href="https://ufos-api.microcosm.blue/">UFOs</a>, <a href="https://spacedust.microcosm.blue/">Spacedust</a>, and <a href="https://tangled.org/microcosm.blue/">more</a>. Most of it runs on hardware that would embarrass a hyperscaler (might be <a href="https://bsky.app/profile/bad-example.com/post/3mabnbs3vbs2i">a Raspberry Pi</a> or three involved), and it’s all free to use thanks to direct <a href="https://ko-fi.com/bad_example">community</a> <a href="http://github.com/sponsors/uniphil/">sponsorship</a> helping cover costs.</p>
<p>Hubble fits right in with these projects! I’m so excited to keep building and being part of the growing community of independent infrastructure developers and operators in the Atmosphere.</p>
<p>More PDSes come online every week, and more people are using them! Even the number of different PDS <em>implementations</em> seems to grow by the day.</p>

<p>This diversification is really good news for the future of atproto!</p>
<p><em>And also</em>, to be realistic about it: some of these PDSes might not have robust backup plans, or the people running them might get bored, or a new implementation might have bugs, and all of this might lead to more PDSes to going offline. Maybe one day a PDS will even try to stop their users from moving away.</p>
<p>Hubble is one small step to counter these effects, in line with the <a href="https://atproto.com/articles/atproto-ethos">atproto ethos</a>. It fills the gap left by relays switching to non-archival mode; it’s a new baseline for account recovery and even <a href="https://www.da.vidbuchanan.co.uk/blog/adversarial-pds-migration.html#repo-backups">adversarial PDS migration</a>; and it’s a data availability backstop for developers building apps.</p>
