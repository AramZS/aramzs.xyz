<div><div data-rw-toc-level="1" data-rw-toc-title="A harness for every task: dynamic workflows in Claude Code "><p><figure><img alt="A harness for every task: dynamic workflows in Claude Code " src="https://pbs.twimg.com/media/HJ0q6o6aYAEM_ej.jpg"/></figure></p><p>Last week, we released <a href="https://code.claude.com/docs/en/workflows" rel="nofollow">dynamic workflows</a> in Claude Code. Claude can now write its own <a href="https://code.claude.com/docs/en/glossary#agentic-harness" rel="nofollow">harness</a> on the fly, custom-built for the task at hand.  </p><p>While the default Claude Code harness is built for coding, it is also useful for many other types of tasks because, as it turns out, many tasks resemble coding tasks. But there are certain classes of tasks where we have had to build custom harnesses on top of Claude Code to achieve peak performance such as <a href="https://support.claude.com/en/articles/11088861-using-research-on-claude" rel="nofollow">Research</a>, <a href="https://support.claude.com/en/articles/11932705-automated-security-reviews-in-claude-code" rel="nofollow">security analysis</a>, <a href="https://code.claude.com/docs/en/agent-teams" rel="nofollow">agent teams</a>, or <a href="https://code.claude.com/docs/en/code-review" rel="nofollow">Code Review</a>.  </p><p>Workflows allow you to dynamically create harnesses that enable Claude to solve all of those problems and more natively inside of Claude Code. You can also share and re-use these workflows with others.  </p><p>In this article, I’ll cover my initial workflows experiences and learnings so you can take full advantage.   </p><p>That said, best practices are still developing! Dynamic workflows often use more tokens, so think carefully about when and how to use them.</p><p><strong>Note</strong>: this post is also <a href="https://claude.com/blog/a-harness-for-every-task-dynamic-workflows-in-claude-code" rel="nofollow">available on the Claude Blog</a> </p><h2>Example prompts</h2><p>Before diving into the technical details, I’d like to start with some example prompts to get you thinking about the possibilities with workflows:  </p><ul><li>"This test fails maybe 1 in 50 runs. Set up a workflow to reproduce it, form theories and adversarially test them in worktrees /goal don't stop until one theory works."  </li><li>"Using a workflow, go through my last 50 sessions and mine them for corrections I keep making and turn the recurring ones into <a href="http://CLAUDE.md" rel="nofollow">CLAUDE.md</a> rules"  </li><li>“Use a workflow to dig through #incidents in Slack for the past six months and find recurring root causes where nobody has filed a ticket."  </li><li>"Take my business plan and run a workflow where different agents tear it apart from an investor's, a customer's, and a competitor's perspective."  </li><li>"Here's a folder of 80 resumes, use a workflow to rank them for the backend role and double-check the top ten. Interview me using the AskUserQuestion tool for a rubric."  </li><li>"I need a name for this CLI tool. Use a workflow to brainstorm a bunch of options and run a tournament to pick the top 3."  </li><li>"Use a workflow to rename our User model to Account everywhere."  </li><li>“Go through my blog post draft and using a workflow verify every technical claim against the codebase, I don't want to ship anything wrong."  </li></ul><h2>How dynamic workflows work</h2><p>Dynamic workflows execute a javascript file with a few special functions that help spawn and coordinate <a href="https://code.claude.com/docs/en/sub-agents" rel="nofollow">subagents</a>:</p><p><figure><img alt="Image" src="https://pbs.twimg.com/media/HJ0t9PDbQAAYqh-.jpg"/></figure> </p><p>Dynamic workflows also include standard JavaScript functions like JSON, Math, and Array, to help process data.  </p><p>It’s particularly useful to know that dynamic workflows can decide which models an agent uses and whether subagents are run in their own worktree, allowing Claude to choose the intelligence level and isolation needed.  </p><p>If a workflow is interrupted, for example by user action or quitting the terminal, resuming the session will allow the workflow to pick up where it left off.  </p><h2>Why dynamic workflows</h2><p>When you ask the default Claude Code harness to do a task, it needs to both plan and execute in the same context window. For many coding tasks, this is highly effective, but it can sometimes break down over long-running, massively parallel and/or highly structured adversarial tasks.  </p><p>This is because the longer Claude works on a complex task in a single context window, the more it becomes susceptible to a few specific failure modes:  </p><ul><li><strong>Agentic laziness</strong> refers to when Claude stops before finishing a particularly complex, multi-part task and declares the job done after partial progress, for example addressing 20 of the 50 items in a security review.  </li><li><strong>Self-preferential bias</strong> refers to Claude’s tendency to prefer its own results or findings, especially when asked to verify or judge them against a rubric.  </li><li><strong>Goal drift</strong> refers to the gradual loss of fidelity to the original objective across many turns, especially after compaction. Each summarization step is lossy, and details like edge-case requirements or "don't do X" constraints can get lost.  </li></ul><p>Creating a workflow helps combat these by orchestrating separate Claudes with their own context windows and focused, isolated goals.  </p><h2>Dynamic vs static workflows</h2><p>You may have previously created a static workflow using the Claude Agent SDK or claude -p to coordinate multiple instances of Claude Code together.  </p><p>But because static workflows need to work for all edge cases, they are usually more generic. With <a href="https://www.anthropic.com/news/claude-opus-4-8" rel="nofollow">Claude Opus 4.8</a> and dynamic workflows, Claude is now intelligent enough to write a custom harness tailor-made for your use case.</p><p><figure><img alt="Image" src="https://pbs.twimg.com/media/HJ1Dc3waUAA8Eeo.png"/></figure> </p><h1>Helpful patterns when using dynamic workflows</h1><p>You can start using dynamic workflows just by asking Claude to make one, or by using the trigger word “ultracode” to ensure that Claude Code creates a workflow.  </p><p>But building a mental model for how dynamic workflows work will help you understand when to use them and how you might nudge Claude via prompts.  </p><p>There are a few common patterns that Claude might use and compose together when building workflows:</p><p><figure><img alt="Image" src="https://pbs.twimg.com/media/HJ0u_2cbMAA3ufP.jpg"/></figure> </p><p><strong>Classify-and-act</strong> </p><p>Use a classifier agent to decide on the type of task, and then route to different agents or behavior based on the task. Or, use a classifier at the end to determine output.  </p><p><strong>Fan-out-and-synthesize</strong> </p><p>Split up a task into many smaller steps, run an agent on each step and then synthesize those results. This is particularly useful for when there are a large number of smaller steps, or when each step benefits from its own clean context window so they don't interfere or cross-contaminate. The synthesize step is a barrier—it waits for all the fan-out agents, then merges their structured outputs into one result.  </p><p><strong>Adversarial verification</strong> </p><p>For each spawned agent, run a separate spawned agent to adversarially verify its output against a rubric or criteria.  </p><p><strong>Generate-and-filter</strong> </p><p>Generate a number of ideas on a topic and then filter them by a rubric or by verification, dedupe duplicates and return only the highest quality, tested ideas.  </p><p><strong>Tournament</strong> </p><p>Instead of dividing the work, have agents compete on it. Spawn N agents that each attempt the same task using different approaches. Prompts or models then judge the results in a pairwise fashion using a judging agent until you have a winner.  </p><p><strong>Loop until done</strong> </p><p>For tasks with an unknown amount of work, loop spawning agents until a stop condition is met (no new findings, or no more errors in the logs) instead of a fixed number of passes.  </p><h1>Use cases</h1><p>Think creatively of when and how to ask Claude Code to make dynamic workflows. I’ve found that workflows are sometimes even more useful for non-technical work.</p><p><figure><img alt="Image" src="https://pbs.twimg.com/media/HJ018ZjbcAA4nFm.png"/></figure> </p><h2>Migrations and refactors</h2><p><a href="https://bun.com/" rel="nofollow">Bun</a> was rewritten from Zig to Rust using workflows. You can read more about how that was done in <a href="https://x.com/jarredsumner/status/2060050578026189172" rel="nofollow">Jarred’s X thread</a>.  </p><p>The key is to break down the task into a series of steps that need to be operated on for example callsites, failing tests, modules, etc. Spin off a subagent for every fix in a worktree to make the fix, then have another agent adversarially review, and merge them. Consider telling the agent not to use resource intensive commands so that you can maximally parallelize without running out of resources on your machine.  </p><h2>Deep research</h2><p>We published a deep research skill (/deep-research) inside Claude Code that uses dynamic workflows. Specifically, it fans-out web searches, fetches sources, adversarially verifies their claims, and synthesizes a cited report.  </p><p>But you may do this sort of research for more than just web searches. For example, asking Claude to compile a status report from context in Slack or to research how a feature works by exploring a codebase in-depth.  </p><h2>Deep verification</h2><p><figure><img alt="Image" src="https://pbs.twimg.com/media/HJ0vMJ8acAAM9Uo.jpg"/></figure> </p><p>On the other hand, if you have a report where you want to check and source every factual claim that it references you may want to generate a workflow which has one agent identify all of the factual claims and then spin off a subagent to check each one in-detail. You could also have a verification agent check the source subagent to make sure its source is high quality.  </p><h2>Sorting</h2><p><figure><img alt="Image" src="https://pbs.twimg.com/media/HJ0wACFbMAAAvWl.jpg"/></figure> </p><p>You may have a list of items that you want to sort by some qualitative measurement that you believe that Claude Code is good at evaluating, for example: support tickets sorted by severity of the bug. But if you try to sort 1000+ rows in one prompt, quality degrades and it won't fit in context. Instead run a tournament, a pipeline of pairwise-comparison agents (comparative judgment is more reliable than absolute scoring), or bucket-rank in parallel then merge. Each comparison is its own agent, so the deterministic loop holds the bracket and only the running order stays in context.  </p><h2>Memory and rule adherence</h2><p><figure><img alt="Image" src="https://pbs.twimg.com/media/HJ0wF76acAAyRMo.jpg"/></figure> </p><p>If you have a particular set of rules that you find Claude misses or struggles with, even when put into the CLAUDE.mds, create a workflow with a list of rules that must be checked by verifier agents—one verifier per rule. Creating a skeptic persona subagent to review the rules to make sure they are in line will help avoid too many false positives.  </p><p>The reverse direction works too: mine your recent sessions and code review comments for corrections you keep making, cluster them with parallel agents, adversarially verify each candidate (would this rule have prevented a real mistake?), and then distill the survivors back into a <a href="http://claude.md/" rel="nofollow">CLAUDE.md</a>.  </p><h2>Root-cause investigation</h2><p>Debugging works best when you come up with several independent hypotheses and test them, but if you’re only using one context window, Claude can run into self-preferential bias.</p><p>A workflow can structurally prevent this by spinning up agents to generate hypotheses from disjoint evidence. For example, separate agents for logs, files, and data. Each hypothesis can then face a panel of verifiers and refuters.  </p><p>This isn't just for code. Workflows can be used for sales (why did sales drop in March?), data engineering (why did this pipeline fail?), or any post-mortem exercise.  </p><h2>Triaging at scale</h2><p><figure><img alt="Image" src="https://pbs.twimg.com/media/HJ0wNb8bUAA33h5.jpg"/></figure> </p><p>Every team has a support queue, bug reports, or some other backlog that cannot be fully processed by humans.  </p><p>A triage workflow classifies each item, dedupes against what's already tracked, and takes action. This could mean attempting the fix or escalating to a human user.  </p><p>A useful pattern for triage workflows is quarantine. This involves barring the agents that read untrusted public content from taking high-privilege actions, which are instead done by the agents in charge of acting on the information.  </p><p>Pair triage workflows with /loop to have Claude do this continuously.  </p><h2>Exploration and taste</h2><p>Workflows can be useful when exploring different approaches to a solution, especially when it is taste based, like design or naming, and would benefit from a rubric.  </p><p>Try asking Claude to explore a bunch of solutions, and give a review agent  a rubric for what a good solution looks like. The task is complete when the review agent feels like it has met the criteria. Solutions can also be ordered or selected via a tournament based on the rubric.  </p><h2>Evals</h2><p>You can run lightweight evals for particular tasks by spinning off separate agents in a worktree and then spinning off comparison agents to compare and grade the specific outputs against a rubric. For example, evaluating and then refining a skill you’ve created against a particular criteria.  </p><h2>Model and intelligence routing</h2><p>Create a classifier agent tuned to your tasks that decides which model to use. This can be helpful when your task will involve many tool calls and conducting research prior to execution can identify the best model for the job.  </p><p>For example, the best model for the task “explain how the auth module works” depends on how many files in the auth module there are and the shape of the codebase. A classifier agent can do this research and then route to Sonnet or Opus based on the expected complexity of the task.  </p><h2>When not to use dynamic workflows</h2><p>Workflows are new. While there are many use cases where it will create outsized results, they are not needed for every task and may end up using significantly more tokens.  </p><p>It’s best to use workflows creatively to push Claude Code in ways that you haven’t previously. For regular coding tasks, try and ask yourself does it really need more compute? For example, most traditional coding tasks do not need a panel of 5 reviewers.  </p><h1>Tips for building dynamic workflows</h1><p><strong>Prompting</strong> </p><p>Detailed prompting, using the specific techniques we described above, for dynamic workflows creates the best results.  </p><p>Workflows are not just for large tasks. You can prompt the model to use a “quick workflow.” For example, you can create a quick adversarial review of an assumption.  </p><p><strong>Combine with /goal and /loop</strong> </p><p>When using workflows that can be repeated, for example triage, research, or verification, pair them with /loop to be run at regular intervals, and /goal to set a hard completion requirement.  </p><p><strong>Token usage budgets</strong> </p><p>You can set explicit token usage budgets for dynamic workflows to limit how many tokens a task uses. You can prompt it with a budget like: “use 10k tokens,” which will set the cap.  </p><p><strong>Saving and sharing dynamic workflows</strong> </p><p>You can save workflows by pressing “s” in the workflow menu. You can check these into ~/.claude/workflows or distribute them via a skill.</p><p><figure><img alt="Image" src="https://pbs.twimg.com/media/HJ0wWqbasAAcvgu.jpg"/></figure> </p><p>To share them via a skill, put your JavaScript workflow files in the skill and folder and reference them in the <a href="http://skill.md/" rel="nofollow">SKILL.MD</a>. To allow for more flexibility, you may want to prompt Claude to think of the workflows in the skill as a template instead of a script that needs to be run verbatim.</p><p><figure><img alt="Image" src="https://pbs.twimg.com/media/HJ0wbLFbIAAu2FH.jpg"/></figure> </p><h2>A whole new world</h2><p>Workflows are a helpful new way to extend Claude Code. I encourage you to think of this as a starting point, there's still much to discover in how to use them best. Let us know what you find.  </p><p><em>Thariq Shihipar and Sid Bidasaria (@sidbid) are members of technical staff at Anthropic, working on Claude Code.</em> </p></div></div>
