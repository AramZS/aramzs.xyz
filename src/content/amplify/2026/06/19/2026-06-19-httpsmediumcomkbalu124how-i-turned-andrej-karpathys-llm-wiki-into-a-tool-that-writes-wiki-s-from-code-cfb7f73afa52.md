<html><body><p><strong>By Balu Kosuri</strong></p><p>A couple of days ago I wrote an article about LLM Wiki, Andrej Karpathy’s pattern for turning piles of documents into a self-maintaining wiki. In the comments, someone asked whether the same idea could ingest code repositories instead of documents. That question stuck with me, and this project is what came out of it.</p><p><strong>What it is:</strong> a small template that points the LLM Wiki pattern at a code repo and generates documentation from it: end-user docs, internal docs, whatever template you define.</p><p><strong>How it works:</strong> you clone the repo, edit one config file, and run one install script. After that, every <code>git commit</code> triggers an agent in the background that reads the diff, updates the wiki, and keeps the docs in sync with the code.</p><figure><img height="820" src="https://miro.medium.com/v2/resize:fit:1400/1*UlkuBC7quC2yskt9AKmKHA.png" width="1232"/></figure><p><em><strong>My Repository: </strong><a href="https://github.com/balukosuri/docs-from-with-llm-wiki-karpathy"><code>https://github.com/balukosuri/docs-from-code-with-llm-wiki</code></a></em>
<em><strong>My Previous article: </strong><a href="https://medium.com/@balukosuri/i-used-karpathys-llm-wiki-to-build-a-knowledge-base-that-maintains-itself-with-ai">I used Karpathy's LLM Wiki to build a knowledge base that maintains itself with AI</a></em>
<em><strong>Original idea by Andrej Kaarpathy:</strong> <a href="https://gist.github.com/karpathy/1dd0294ef9567971c1e4348a90d69285">Karpathy's llm-wiki.md</a></em></p><p>The rest of this article is the story of why it works and where it breaks.</p><h3>What this project is, and what it is not</h3><p>I want to be blunt about this because it’s the question that gets asked (and answered wrongly) most often.</p><p><strong>This is a small personal project.</strong> It instantiates one of Andrej Karpathy’s ideas (the <code>llm-wiki.md</code> pattern) and points it at a code repository instead of a document folder. Generating wiki's from code</p><p><strong>The problem it addresses</strong> is specific: internal wikis and engineering knowledge bases die on the vine. They start well, drift out of date because maintenance is nobody’s job, and end up distrusted. A README that’s 20% wrong is worse than no README, because now people are confidently misinformed.</p><p><strong>This is not a replacement for a technical writer.</strong> I want to say that twice because it’s important and because I am a technical writer. A writer brings judgment, narrative, audience empathy, editorial voice, and the willingness to push back on the engineers they work with. An LLM has none of those. What this tool produces is:</p><ul><li>a <strong>draft</strong> a writer can polish, not a finished doc</li><li>an <strong>internal reference</strong> for developers, not a published product manual</li><li>a <strong>bookkeeper</strong> for the 80% of content that decays as code changes, freeing human writers for the 20% that needs voice</li></ul><p>If you have a writer on your team, this tool helps them by giving them a living, code-grounded reference to work from. It does not replace them. If you do not have a writer, this tool gives you something your team can trust for internal use, but do not mistake it for customer-ready documentation without a human in the loop.</p><p><strong>Best fit:</strong> internal developer docs, CLI tools and libraries, solo devs and small teams without a docs function, teams that want a living reference alongside their human-authored product docs.</p><p><strong>Poor fit without a human editor:</strong> shipped end-user product documentation, UI-heavy apps where behaviour depends on runtime data, compliance / legal / safety-critical content.</p><p>With that disclaimer out of the way, here’s why I built it.</p><h3>Why code documentation always goes out of date</h3><p>Think about every README you’ve ever written. It started strong. Someone onboarded, skimmed it, and shipped a fix. A month later a feature got renamed. The README didn’t. Three months later a whole subsystem was rewritten. The README didn’t change. Six months later the README is a relic: wrong enough to mislead, right enough that nobody deletes it.</p><p>This is not a motivation problem. It is a <strong>timing</strong> problem.</p><p>Documentation decays because the moment the code changes is not the moment the docs get updated. By the time anyone notices, the developer who made the change has context-switched away and now lives in another world.</p><p>For my previous article this was easy to solve for static documents. You just “ingest” a PDF once and you’re done. But code is not static. Every commit is a potential ingest. Nobody wants to type <code>ingest src/auth/login.ts</code> every time they save a file.</p><p>So I needed a different trigger.</p><h3>Why git commit is the right trigger for updating documentation</h3><p>Here is the move. The natural heartbeat of a code project is not “when the developer remembers to update docs”. It is <strong><code>git commit</code></strong>. Every commit is an atomic unit of change: tested, reviewed (hopefully), and marked with a human-written subject. If the wiki updates itself on every commit, the docs never fall more than one commit behind the code.</p><p>That is what <code>docs-from-code-with-llm-wiki</code> does. A <code>post-commit</code> git hook fires in the background. It diffs <code>HEAD~1..HEAD</code>, hands the diff to whichever AI tool you configured, and the CLI updates the wiki in place. A few seconds later a follow-up commit lands with the subject <code>wiki: update (&lt;sha&gt;)</code>. Your <code>git commit</code> returned instantly. You kept coding.</p><p>Here is the whole flow:</p><pre><code>git commit (code)
   │
   ▼
post-commit hook → nohup ingest.sh &amp;   (returns immediately, you keep coding)
                          │
                          ├─ flock (prevents overlapping runs)
                          ├─ git diff &lt;last-ingested&gt;..HEAD
                          ├─ claude -p  (or cursor-agent, or codex)
                          │     └─ reads CLAUDE.md + config.yml
                          │        edits wiki/*.md in place
                          └─ git commit -m "wiki: update (&lt;sha&gt;)"</code></pre><p>For a vibe coder, someone who codes in flow, commits when they feel like it, and does not want to be a documentation bureaucrat, this is the ideal loop. You never type “ingest”. You just commit. The docs appear.</p><h3>What kind of documentation this AI wiki actually produces</h3><p>The second question, after “what triggers it”, is “what does it produce”. And here I did not want to make one opinionated choice. Internal architecture docs, end-user README drafts, API reference, ADRs. Different projects want different mixes. Some want all of them; some want none.</p><p>So the template has <strong>one configuration file</strong> that flips categories on and off:</p><pre><code># .llmwiki/config.yml</code></pre><pre><code>cli: claude              # claude | cursor-agent | codex</code></pre><pre><code>doc_types:
  architecture: true     # Internal: module map, data flow
  api: true              # Public function / class / command reference
  user: false            # End-user README drafts and usage pages
  decisions: true        # ADR-style decision records
  concepts: true         # Domain ideas discovered in the code</code></pre><pre><code>include:
  - "src/**"
  - "lib/**"</code></pre><pre><code>exclude:
  - "**/*.test.*"
  - "dist/**"</code></pre><pre><code>custom_types: []
#  - name: runbook
#    dir: runbooks/
#    trigger: "when the diff touches infra/ or ops/"</code></pre><p>Turn on what you care about. Add your own categories under <code>custom_types</code>. That's the only knob.</p><p>On every ingest, the CLI re-reads this file. If you had <code>user: false</code> yesterday and flip it to <code>true</code> today, the next commit will start drafting end-user docs. No code changes required.</p><p>The enabled categories map one-to-one to folders under <code>wiki/</code>:</p><pre><code>wiki/
  index.md          ← master catalog
  log.md            ← append-only activity log (great for `git blame` of docs)
  overview.md       ← big-picture synthesis
  glossary.md       ← every public identifier, flag, config key
  architecture/     ← per-subsystem pages (grouped by responsibility)
  api/              ← one page per exported function, class, or command
  user/             ← README drafts, how-tos, usage pages
  decisions/        ← ADR pages proposed from commit messages
  concepts/         ← domain ideas (e.g. "retry backoff", "job queue")
  sources/          ← one summary page per significant source file</code></pre><p>Open it in Obsidian and the graph view colour-codes by folder. You can see at a glance which subsystems are hubs and which are orphans.</p><h3>How to stop the LLM from hallucinating about your code</h3><p>Code is an ungrateful source to summarize.</p><p>Code narrates <em>what</em> happens, like <code>function login(user, pass)</code>. It almost never narrates <em>why</em> the function exists, who calls it, what happens when it fails at 3am in production, or what the user sees when it does. An LLM summarizing a function without that context will happily fill in the gaps from its training distribution. If the code looks like other code it has seen, it will describe the behaviour the same way, even when the specific implementation does something unique.</p><p>How bad this is depends on how much of the behaviour actually lives in the code. Some code surfaces are readable end-to-end from the source: a function signature, a typed API schema, a config definition. Other code surfaces depend heavily on things the source does not narrate: styling, runtime data, user interaction, environment. The more behaviour lives outside the code, the more the AI has to guess, and the more it will hallucinate.</p><p>There is no silver bullet for this, but two layered defences cover most of the damage:</p><h3>1. Prompt discipline with a strict CLAUDE.md file</h3><p><code>CLAUDE.md</code> is the schema file that every AI CLI reads before it does anything. The version that ships with this template has seven hard rules. The key ones:</p><ul><li><strong>Cite or do not claim.</strong> Every non-trivial statement must be followed by <code>(path:start-end)</code>. If you can't produce the citation, you are speculating. Stop and emit a <code>&gt; TODO-VERIFY:</code> blockquote instead.</li><li><strong>Never describe an API that is not at HEAD.</strong> Don’t write “this function accepts an optional <code>timeout</code>" if <code>timeout</code> is not in the current signature.</li><li><strong>Never narrate runtime behaviour you cannot read.</strong> No “this probably does X”. If behaviour is only inferable from tests, cite the test file.</li><li><strong>UI clamp.</strong> For React components, templated HTML, and CSS-driven behaviour, do not describe what the user sees unless a test, snapshot, or storybook file confirms it.</li><li><strong>Never cite anything outside this repo.</strong> No external docs, no “frameworks like this usually…”. Only code at HEAD.</li></ul><p>The CLI is told, in loud uppercase, that <code>TODO-VERIFY</code> blocks are a feature, not a failure. A page with two confident paragraphs and three <code>TODO-VERIFY</code> markers is more useful than a page covered in confident hallucinations.</p><h3>2. A freshness check backed by real git SHAs</h3><p>Every wiki page stores the <code>git hash-object</code> SHA of each file it cites, right there in the frontmatter:</p><pre><code>---
title: Login flow
type: module
sources:
  - path: src/auth/login.ts
    sha: 3f9a21bccd4e0f12abc34...
    lines: 1-120
  - path: src/auth/session.ts
    sha: 0b12aa99cc8d77eeff01...
    lines: 15-88
updated: 2026-04-17
---</code></pre><p>The freshness script walks every page, re-runs <code>git hash-object</code> on each cited file, and reports any page whose citations no longer match. Two things make this cheap and reliable:</p><ul><li>Git blob SHAs are deterministic. Same content → same SHA. Always.</li><li>You can run it in a pre-push hook, a CI job, or manually with <code>bash .llmwiki/freshness.sh</code>. It reads nothing but the working tree.</li></ul><p>When a page is stale, you paste the stale list to your AI and say “re-generate these”. The next ingest has real code in front of it, and the hallucinated claims go away.</p><p>Is this bulletproof? No. Nothing is. But it turns hallucinations from a silent, cumulative decay problem into a loud, dated, traceable one. You always know which pages might be wrong, and why.</p><h3>How this differs from the original LLM Wiki for documents</h3><p>Side-by-side comparison with the original technical-writer version I shipped two weeks ago:</p><figure><img height="844" src="https://miro.medium.com/v2/resize:fit:1400/1*yXjDXW34ojSnZ7oeTkCNfg.png" width="1486"/></figure><p>Same pattern, different dials.</p><h3>What using this looks like in a real developer workflow</h3><p>Imagine you’re building a small project over the weekend. You don’t want to stop and write docs. You just want to code.</p><p>You install the template once and forget about it. From that point on, every time you run <code>git commit</code>, a small agent runs in the background, reads what changed, and updates the wiki for you.</p><ul><li>You add a new function or endpoint → a page for it shows up in the wiki.</li><li>You change a signature or config option → the relevant page updates itself.</li><li>You rename or split a file → the wiki reshuffles itself to match.</li><li>Something doesn’t line up between the code and the docs → the wiki leaves a <code>TODO-VERIFY</code> note pointing at the mismatch, so you can catch it before shipping.</li></ul><p>You never open the wiki while coding. You just commit. Later, when you want to share your project, write a blog post, or onboard someone, you open the <code>wiki/</code> folder and most of the documentation is already there: overview, reference, glossary, and a graph of how the pieces connect.</p><p>The mental model is simple: <strong>you write code, the agent writes docs, git is the handoff between the two.</strong></p><h3>The honest limitations of AI generated code documentation</h3><ul><li><strong>Hallucinations are mitigated, not eliminated.</strong> The citation discipline in <code>CLAUDE.md</code> and the SHA-based freshness check reduce drift. They do not guarantee correctness. Any AI-authored claim can still be wrong.</li><li><strong>UI-heavy code</strong> is the weakest case. If you work on a React/Vue/Svelte app, turn <code>api</code> and <code>architecture</code> on, turn <code>user</code> off, and treat the output as a skeleton a human has to finish. Do not publish the generated end-user docs without review.</li><li><strong>Large diffs</strong> (massive refactors, rename-everything commits) get skipped by default above 2000 lines. You can lift the limit, but expect cost. The template prints a skip message rather than silently burning tokens.</li><li><strong>Renames across files.</strong> Git’s rename detection is heuristic. If the AI sees “delete foo.ts + add bar.ts” it may not realise they are the same thing. The hook passes the commit subject through, so a subject like <code>refactor: rename foo.ts to bar.ts</code> gives it a strong hint.</li><li><strong>Every commit costs money.</strong> The hook calls an AI API on every matching commit. Tighten the globs and turn off unused doc types to keep the bill sane.</li><li><strong>Not a single source of truth.</strong> If you also have Confluence / Notion / a docs site, this sits alongside them. It documents the code-grounded slice; the rest stays where it is.</li><li><strong>Private APIs vs public APIs.</strong> The template defaults to documenting the public surface. If you want internals documented too, broaden the <code>include</code> glob and enable <code>architecture</code>.</li><li><strong>Monorepos.</strong> Works fine with one wiki per repo. For a monorepo, either run one wiki in the repo root (broad) or put a separate <code>.llmwiki/</code> and <code>wiki/</code> per package (focused). The latter tends to be more useful.</li><li><strong>CI-only workflows.</strong> This template runs locally. Teams that want the wiki updated on merge-to-main rather than per-commit-per-developer should adapt the hook into a GitHub Action. The prompts and <code>CLAUDE.md</code> port straight over.</li><li><strong>Windows.</strong> Tested on macOS. Linux should work. Windows requires Git Bash or WSL; native cmd / PowerShell will not run the hook.</li><li><strong>Quality tracks the CLI.</strong> The wiki is only as good as the model you put behind it. Upgrade the model, upgrade the wiki.</li><li><strong>Not a reviewer.</strong> This documents code. It does not tell you the code is wrong, insecure, or could be simpler.</li></ul><h3>Common misconceptions about AI code documentation</h3><p>A handful of misinterpretations worth clearing up in one place:</p><ul><li><strong>“This replaces technical writers.”</strong> It does not. It produces drafts and a living reference. The last mile (voice, narrative, audience empathy, editorial judgement) is still a human job.</li><li><strong>“I can publish the generated output as end-user docs.”</strong> For internal use, often yes. For customer-facing content, always review. The hallucination risk on behaviour is real.</li><li><strong>“It understands the whole codebase.”</strong> It understands the diff and the pages that cite the changed files. For a full sweep, ask your CLI directly to “re-read <code>wiki/index.md</code> and deep-scan". That's a different operation.</li><li><strong>“No <code>CONTRADICTION</code> block means the wiki is correct."</strong> Absence of flagged issues is not absence of issues. Run <code>freshness.sh</code> and spot-check the cited line ranges. Trust, but verify.</li><li><strong>“The hook fixes bad code.”</strong> It documents whatever is there. It is not a reviewer or a refactor tool.</li><li><strong>“One commit updates one wiki page.”</strong> One commit often touches 5–15 pages: glossary, index, overview, plus the affected entity pages. That is the design.</li><li><strong>“Turning on every doc type gives me better docs.”</strong> It gives you broader docs at higher cost. Narrow, enabled categories produce sharper pages.</li><li><strong>“My team can use this instead of Confluence.”</strong> Different tool, different purpose. This lives next to the code. Product specs, org knowledge, and customer-facing docs belong in their existing homes.</li><li><strong>“A <code>wiki: update</code> commit that looks right means the wiki is right."</strong> It means the AI thought it was right. Periodic human review is still required.</li></ul><p>If you catch yourself thinking any of the left-hand versions, come back to this list.</p><h3>How to install the LLM Wiki in your repo in a few minutes</h3><pre><code>git clone https://github.com/balukosuri/docs-from-code-with-llm-wiki.git my-project
cd my-project</code></pre><pre><code># Edit .llmwiki/config.yml: pick your CLI, toggle doc types
$EDITOR .llmwiki/config.yml</code></pre><pre><code># Install the hook
bash .llmwiki/install-hook.sh</code></pre><pre><code># Commit some code (even "initial commit" works)
git add . &amp;&amp; git commit -m "start"</code></pre><pre><code># Watch the wiki populate
tail -f .llmwiki/state/ingest.log</code></pre><p>Open the folder in Obsidian as a vault. The overview page opens by default. Graph view is <code>Cmd+G</code>.</p><p><strong>If you are vibe coding</strong>, the whole thing is: clone, edit one YAML file, run one script, then forget the wiki exists. Your commits write it for you.</p><h3>Final thoughts</h3><p>Documentation failures are almost never “we didn’t know how to write docs”. They are “the moment the code changed and the moment the docs should have changed are separated by too much time and too many context switches”.</p><p>Karpathy’s LLM Wiki pattern collapses that gap for documents by making an AI agent the wiki maintainer. This variant collapses it for code by making <code>git commit</code> the trigger and grounding every claim in <code>path:line</code> citations that a freshness check can verify.</p><p>The wiki is not a thing you maintain alone. It’s a thing the hook drafts and you refine. Your job shifts from bookkeeper to reviewer: write the code, commit it, and review what the hook proposes. The authoring that actually matters (voice, narrative, judgement, editorial taste) is still yours. The tool just removes the grunt work that used to stand between you and the parts of documentation that need a human.</p><p>If the previous article was about turning 15 scattered documents into a self-updating knowledge base, this one is about turning a live, evolving codebase into its own living documentation. The pattern is the same. The trigger is what changed.</p><p>My name is Balasubramanyam Kosuri, and I work as a technical writer.
