<html><body><h3>Building an LLM Research Wiki: How I Turned 3,000 Pages of Philosophy into a Living Knowledge System</h3><p><em>How an artist-researcher adapted Andrej Karpathy’s LLM Wiki pattern to build a structured knowledge base across Continental philosophy, music ontology, and posthumanism — using Claude Code as a dedicated research intelligence agent.</em></p><p>—</p><figure><img height="1374" src="https://miro.medium.com/v2/resize:fit:1400/1*yIc1Tyhlj3RhS1AciawJvw@2x.jpeg" width="1814"/><figcaption>The open-sourced <a href="http://github.com/MetamusicX/llm-research-wiki">LLM Research Wiki</a> repository on GitHub.</figcaption></figure><h4>The Spark</h4><p>In early April 2026, Andrej Karpathy published a <a href="https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f">gist</a> describing what he called an “LLM Wiki” — the idea that instead of using RAG (Retrieval-Augmented Generation) to re-derive knowledge from raw documents on every query, an LLM should incrementally build and maintain a <strong>persistent, structured wiki</strong> that compounds over time. Raw sources go in; structured, interlinked knowledge comes out. The wiki becomes smarter than your memory about connections across your reading.</p><p>I was immediately interested. As an artist-researcher working across Continental philosophy, music composition, and posthumanism, I deal with dense, interconnected primary sources — Deleuze, Simondon, Rancière, Stiegler, Barad — where the connections between concepts across books and authors are precisely what makes the research productive. Traditional note-taking and reference management tools don’t capture these connections. They store documents; they don’t <em>think across</em> them.</p><p>Karpathy’s pattern was designed for a software engineer’s needs (Obsidian-based, focused on technical documentation). I needed something for a humanities researcher working with 300-page philosophical monographs where a single concept might appear in five different authors with five different meanings. So I set out to adapt it.</p><h4>The Design Phase</h4><p>I started by taking Karpathy’s core insight — raw sources as immutable ground truth, the wiki as a living synthesis layer, structured workflows for ingest/query/lint — and exploring how to adapt it for academic research. I used ChatGPT to brainstorm the initial architecture: What page types does a humanities wiki need? How should concepts, authors, and debates be structured? What kind of frontmatter enables fast navigation?</p><p>This exploration produced the first draft of the schema: six page types (source notes, concepts, authors, debates, syntheses, projects), YAML frontmatter conventions, and the three-layer architecture (raw → wiki → schema).</p><p>Then I moved to Claude Code — Anthropic’s terminal-based coding agent — for the actual implementation. Claude Code operates directly in your file system, reading and writing markdown files, and it retains context across long sessions. This made it the ideal tool for building and maintaining a wiki: it could read a 300-page PDF, create 15 interlinked wiki pages, and update the index — all in a single conversational session.</p><p>An important detail: the initial build was not a single-agent operation. I had previously created three specialised agents within Claude Code, and the setup session orchestrated all three:</p><p><strong>Agent A</strong> — Orchestrated the full setup; routed tasks to the other two agents</p><p><strong>Agent B</strong> — Researched the onboarding strategy for large existing collections (the “5,000 notes / 4,000 PDFs” problem)</p><p><strong>Agent C</strong> — Built the folder structure, authored the CLAUDE.md schema and all template files, and executed the first ingest — which produced 38 wiki pages in a single pass</p><p>This multi-agent division of labour meant that the entire system — schema, folders, templates, onboarding strategy, and a fully populated first ingest — was built in a single session. After that, the agents were no longer needed: the CLAUDE.md schema itself became the permanent operator, read by Claude Code at the start of every subsequent session.</p><p>What I didn’t expect was that Claude Code would become not just the builder but the <em>operator</em> of the wiki. The CLAUDE.md file at the root of the project functions as a permanent instruction set: every time a new session starts, Claude reads it and acts as a dedicated research intelligence agent — following the ingest, query, and lint workflows defined in the schema.</p><h4>The Architecture</h4><p>The system has three layers:</p><p><strong>raw/</strong> — Immutable source documents (PDFs, transcripts, notes). <em>Written by the researcher.</em></p><p><strong>wiki/</strong> — Structured markdown pages (concepts, authors, debates, syntheses, source notes, projects). <em>Written by the LLM.</em></p><p><strong>schema</strong> — CLAUDE.md (operational instructions), index.md (master index), log.md (change log). <em>Written by both.</em></p><p>The raw/ layer is sacred — source files are never modified after being placed there. The wiki/ layer is the living synthesis that grows with every ingest. The schema layer governs everything.</p><h4>Six Page Types</h4><p>Every wiki page follows one of six templates, each with specific YAML frontmatter:</p><ul><li><strong>Source notes</strong> — one per ingested document. Summary, key claims (with page numbers), direct quotes, connections to other pages, open questions.</li><li><strong>Concept pages</strong> — one per concept (e.g., “assemblage,” “transduction,” “haecceity”). Definition, key thinkers, related concepts, source support from multiple texts.</li><li><strong>Author pages</strong> — one per key thinker. Bio sketch, key works, concepts, relevance to my research.</li><li><strong>Debate pages</strong> — framed intellectual disagreements across the literature.</li><li><strong>Synthesis pages</strong> — evolving argumentative overviews across a cluster of related pages.</li><li><strong>Project pages</strong> — active research or writing projects with their concept/source inventories.</li></ul><h4>Epistemic Markers</h4><p>A feature I’m particularly proud of: every claim in the wiki carries an epistemic register marker.</p><ul><li><em>(no marker)</em> — Directly attributable to a named source</li><li><strong>[W]</strong> — Wiki synthesis: the LLM’s editorial integration across multiple sources</li><li><strong>[P]</strong> — My own research position: not what a source says, but what I argue</li><li><strong>[?]</strong> — Uncertain: a date, attribution, or claim the wiki cannot confidently verify</li></ul><p>This matters because in humanities research, the distinction between “what Deleuze says,” “what Sauvagnargues says Deleuze says,” and “what I claim about both” is philosophically consequential. The markers keep these registers visible.</p><h4>Navigation Design</h4><p>After about 50 pages, I hit a scaling problem: a flat alphabetical index becomes too slow to navigate. The solution was a three-layer navigation cascade:</p><ol><li><strong>Concept clusters</strong> in index.md — thematic groupings (4–6 per domain) that a query checks first</li><li><strong>Synthesis pages</strong> — pre-digested argumentative overviews for each cluster (one page instead of six)</li><li><strong>related: YAML fields</strong> — every concept/author page carries 3–5 pointers to its closest neighbours</li></ol><p>The cascade means that at 185 pages, query cost is roughly the same as it was at 50.</p><h4>The Numbers</h4><p>After approximately two weeks of intensive work (April 6–17, 2026), the wiki looks like this:</p><ul><li><strong>Source documents ingested:</strong> 70</li><li><strong>Total wiki pages:</strong> 185 (65 concepts, 39 authors, 70 source notes, 4 debates, 2 syntheses, 5 projects)</li><li><strong>Total cross-references (markdown links):</strong> 1,592</li><li><strong>Total words in wiki:</strong> 233,881</li><li><strong>Pages of primary source material read:</strong> ~3,200</li><li><strong>Log entries:</strong> 73</li></ul><p>The debate and synthesis counts are low — these page types grow more slowly because they require genuine argumentative integration across multiple sources, not just extraction from a single text. They will grow as the wiki matures. The concept and author pages, by contrast, are already dense: every concept page has at least 2 source support entries, and the richest have 17.</p><h4>The Ingest Multiplier</h4><p>On average, each ingested source produces <strong>2.6 wiki pages</strong> (1 source note + updates to ~1.6 existing pages). But this average conceals wide variation:</p><ul><li><strong>A short article</strong> (5–20 pp) typically produces 1 source note + updates to 2–3 existing pages = <strong>3–4 page operations</strong></li><li><strong>A major monograph</strong> (200+ pp) can produce 1 source note + updates to 8–10 existing pages = <strong>10–12 page operations</strong></li><li><strong>The largest single ingest</strong> — Sauvagnargues’s <em>Deleuze and Art</em> (187 pp) — created 7 new pages (6 new concept stubs + 1 source note) and updated 10 existing pages = <strong>17 page operations</strong></li></ul><p>The real power is not in the creation of new pages but in the <em>updating</em> of existing ones. When I ingest Deleuze’s <em>Difference and Repetition</em> Chapter 4, the LLM doesn’t just create a source note — it adds the primary source reference to the <em>multiplicity</em> concept page, updates the <em>differenciation</em> page, enriches the <em>univocal being</em> page, and adds it to the Deleuze author page. Each new ingest makes every previous ingest more valuable.</p><h4>The Densest Nodes</h4><p>Some concept pages have become extraordinarily rich through accumulated ingests:</p><p><strong>Assemblage</strong> — 17 source support entries, spanning Deleuze-Guattari, DeLanda (4 books), Sauvagnargues (2 books), my own texts, Beistegui, Haraway, Rancière, Simondon</p><p><strong>Posthumanism</strong> — 13 entries, spanning Hassan, Haraway, Hayles, Braidotti (2 books), Ferrando (2 sources), Wolfe, Tomlinson, my ERC grant description</p><p><strong>Transduction</strong> — 9 entries, spanning Simondon (2 sources), Stiegler, Hui (2 sources), Beistegui, Sauvagnargues (2 books), my own article on performative transduction</p><p>These densely supported pages are where the wiki becomes genuinely useful as a research tool. The <em>assemblage</em> page, for instance, now contains DeLanda’s properties/capacities distinction, Deleuze-Guattari’s tetravalent definition, Sauvagnargues’s machinic assemblage, my own six musical strata, and the genealogy of the <em>agencement/assemblage</em> translation problem — all in one page, with citations to their primary sources. No single book or article contains all of this. Only the wiki does.</p><h4>The Onboarding Problem</h4><p>If you’re an established researcher, you likely have thousands of notes and thousands of PDFs. The single most important lesson I learned — before ingesting a single source — is this:</p><p><strong>The wiki is NOT the library. It is a curated synthesis of what matters NOW.</strong></p><p>During the design phase, we identified five traps to avoid:</p><ol><li><strong>The Migration Fantasy</strong> — never try to ingest everything. Your existing library stays where it is.</li><li><strong>Premature Categorization</strong> — don’t create 50 empty stubs before ingesting a single source.</li><li><strong>Batch ingesting before the spine exists</strong> — you need 5–10 carefully supervised ingests before the wiki has enough structure to guide itself.</li><li><strong>Starting with your most complex source</strong> — start with your own research map, not with <em>Difference and Repetition</em>.</li><li><strong>Treating raw/ as a copy of your PDF library</strong> — raw/ is a curated intake folder, not a mirror.</li></ol><p>The phased approach that worked for me:</p><ul><li><strong>Phase 0:</strong> Write a research map in your own words — 2–3 pages describing your current research constellation. Ingest it first. This seeds the wiki with <em>your</em> conceptual framework.</li><li><strong>Phase 1 (sources 1–5):</strong> Fully supervised, one source at a time, spread across your projects. You’re training both the wiki and yourself.</li><li><strong>Phase 2 (sources 6–20):</strong> Still supervised, but you can do 2–3 per session. The schema and accumulated wiki context start guiding the LLM’s decisions.</li><li><strong>Phase 3 (sources 20+):</strong> Lighter supervision. Batch ingesting becomes viable. The wiki has enough density to catch its own errors through lint.</li></ul><p>For existing collections: keep your 5,000 notes and 4,000 PDFs where they are. Copy a source to raw/ only when you’re ready to ingest it. For books, ingest chapters rather than entire volumes. The priority test: <em>“Would I cite this in something I’m writing right now?”</em></p><p>The growth trajectory I’ve observed:</p><ul><li><strong>5 sources / ~20 pages</strong> — Basic Q&amp;A on ingested sources</li><li><strong>20 sources / ~80 pages</strong> — First real syntheses viable</li><li><strong>50 sources / ~200 pages</strong> — Draft paper sections from queries</li><li><strong>100 sources / ~400 pages</strong> — Wiki smarter than memory about cross-connections</li></ul><p>The break-even point is around source 20–30: when a query returns a cited answer in 30 seconds instead of 20 minutes of PDF digging.</p><h4>The Three Workflows in Practice</h4><h4>INGEST</h4><p>I say: <em>ingest Deleuze, Gilles. 1968 [1994] Difference and Repetition. CHAPTER 4.pdf</em></p><p>Claude reads the full chapter (54 pages of dense philosophy), then:</p><ol><li>Discusses key takeaways with me before writing anything</li><li>Creates a source note with 13 key claims, 10 direct quotes, and 4 open questions</li><li>Updates the <em>multiplicity</em> concept page (adds the “primary philosophical source”)</li><li>Updates the <em>differenciation</em> page (adds the primary source, demotes Beistegui to “secondary”)</li><li>Resolves the “thin source support” issue on <em>onto-hetero-genesis</em> and <em>univocal being</em></li><li>Updates the Deleuze author page</li><li>Updates index.md and log.md</li></ol><p>Total: 1 page created, 5 pages updated, 2 thin-source issues resolved. Time: about 10 minutes.</p><h4>QUERY</h4><p>I can ask: “How does the line of flight concept work politically?”</p><p>Claude checks index.md, identifies Cluster C, reads the <em>line-of-flight</em> concept page (which has 3 source support entries: ATP Ch. 1, Sauvagnargues 2013, and Thornton 2020), follows the related fields to <em>assemblage</em> and <em>becoming</em>, and constructs an answer from the synthesised wiki — citing specific pages and passages without needing to re-read the original PDFs.</p><h4>LINT</h4><p>I say: <em>lint</em></p><p>Claude audits all 185 pages against 8 criteria: orphan pages, broken links, missing epistemic markers, thin source support, stale pages, source notes not in index, concepts missing from index. The last lint returned: 0 orphans, 0 missing markers, 0 thin-source pages, 0 broken concept links, 1,592 cross-references intact. Every concept page has 2+ sources.</p><h4>What I Learned</h4><h4>1. The wiki is more than the sum of its sources</h4><p>The most valuable pages are the ones that synthesise across sources no single author has connected. The <em>assemblage</em> page — drawing on Deleuze-Guattari’s philosophical concept, DeLanda’s social ontology, Sauvagnargues’s aesthetics, and my own musical application — contains knowledge that exists nowhere else in published form. The wiki <em>produces</em> knowledge through the act of structured accumulation.</p><h4>2. Supervision matters at the start, less so later</h4><p>The first 10–15 ingests required close supervision: checking that concepts were correctly identified, that connections were genuine rather than superficial, that epistemic markers were applied correctly. After that, the schema and the accumulated wiki context guided the LLM toward increasingly accurate and consistent page updates. The wiki trains its own operator.</p><h4>3. The lint workflow is essential</h4><p>Without regular linting, the wiki would drift: orphan pages, broken links, inconsistent markers. The lint workflow catches these before they compound. I run it every 10–15 ingests. It takes 2 minutes and prevents hours of cleanup.</p><h4>4. Obsidian is the natural companion</h4><p>After building the wiki entirely through Claude Code, I opened the folder in Obsidian. Everything worked immediately — the graph view, the backlinks panel, the search. No migration, no conversion. Obsidian reads the same markdown files that Claude writes. The two tools are complementary: Claude for structured operations (ingest, query, lint), Obsidian for visual exploration and serendipitous discovery.</p><h4>5. The schema is the real product</h4><p>The wiki pages are valuable, but the reusable product is the CLAUDE.md schema — the operational instruction set that turns any LLM coding agent into a research wiki operator. It’s <a href="https://github.com/MetamusicX/llm-research-wiki">open-sourced on GitHub</a>. Anyone can clone it, edit the domain context for their own field, and start ingesting.</p><h4>Try It Yourself</h4><p>The repository is at <a href="https://github.com/MetamusicX/llm-research-wiki">github.com/MetamusicX/llm-research-wiki</a>.</p><p>What you need:</p><ul><li><a href="https://claude.ai/code">Claude Code</a> (terminal, desktop, or VS Code extension)</li><li>The CLAUDE.md file from the repo — this is the schema that makes it work</li><li>Your own source documents (PDFs, markdown notes, transcripts)</li><li>No database, no embeddings, no plugins — just markdown files and folders</li></ul><p>Start with your own research map as the first ingest. It seeds the wiki with <em>your</em> conceptual framework. Then add sources one at a time. Supervise the first 10. Run lint every 15. After 50 sources, you’ll have a genuine research tool. After 100, it’s indispensable.</p><p>— -</p><p><em>Paulo de Assis is an artist-researcher with expertise in composition, piano performance, Continental philosophy, science and technology studies, </em>and <em>epistemology.
