<h2 data-endline="1" data-id="Primer" data-startline="1">Primer</h2>
<p>The average Subvert listener probably isn't concerned with metadata from a technical standpoint, but likely <em data-position="119" data-size="0">does</em> want to reliably filter their music library by artist or album release date. Metadata in an abstract sense isn't itself valuable, but having clean structured data will let you build features that are useful for fans and enable music clients to give them the best possible listening experience. Specificaly, because Subvert isn't a streaming platform, metadata is the <em data-position="493" data-size="0">only</em> part of the listening process Subvert actually has the power to influence! The rest is up to the artist.</p>
<h3 data-endline="5" data-id="MusicBrainz-Bandcamp-and-Subvert-A-Metadata-Comparison" data-startline="5">MusicBrainz, Bandcamp, and Subvert: A Metadata Comparison</h3>
<p>The MusicBrainz database is a CC0 licensed open-data project that aims to be a full collection of humanity's published musical works. You can think of it like IMDB for music, except instead of being owned by Amazon, MusicBrainz is stewarded by a non-profit organization and all the data can be downloaded by anybody instead of being locked in IMDB's website and served with a bunch of ads. Alongside the non-profit status and open data licenses, MusicBrainz first launched in July of 2000 making it both a stable architectural choice and a rich data source with over 25 years of user-generated history to draw from. <a href="https://musicbrainz.org/doc/MusicBrainz_Enabled_Applications">A full list of applications that interface with MusicBrainz can be found on their website.</a> If you've ever ripped a CD without using iTunes or played music with VLC there's a decent chance that some of your files have metadata or cover art because of MusicBrainz!</p>
<p>Enthusiasts use MusicBrainz or Discogs to add ID3 metadata tags to their downloaded library of songs as this often enables superior search and filtering experiences than are otherwise made possible with the metadata downloaded from digital storefronts. Shown here is a metadata comparison of Vulfpeck's <em data-position="1916" data-size="0">Half of the Way</em> as tagged with data from Musicbrainz compared against what most fans will ever have, downloaded directly from Bandcamp and Subvert.</p>

<p>These tags notably only include human-readable names as defined by the ID3 spec. While MusicBrainz' database is all UUID-based, the chances that individual user libraries will run into name collisions is reasonably low, but can happen. Unfortunately there is not a widely accepted tagging method that uses both human and machine readable fields for all names.</p>
<p>These tags were extracted with <a href="https://exiftool.org/TagNames/ID3.html">Exiftool, a metadata viewer with solid ID3 tag support</a>. Non-music file metadata related tags have been removed for clarity. The Subvert export was created by uploading an MP3 file with no attached metadata, entering all available fields in the upload form, and downloading the resulting file.</p>
<h2 data-endline="48" data-id="Subverts-Current-Approach" data-startline="48">Subvert's Current Approach</h2>
<p>As part of a release's upload flow, Subvert currently allows artists to enter the following:</p>
<ul data-endline="66" data-position="8285" data-size="0" data-startline="52"> <li data-endline="52" data-position="8287" data-size="0" data-startline="52">Album credits (arbitrary key-value pairs)</li> <li data-endline="53" data-position="8331" data-size="0" data-startline="53">Track credits (arbitrary key-value pairs)</li> <li data-endline="54" data-position="8375" data-size="0" data-startline="54">Label association (arbitrary string)</li> <li data-endline="55" data-position="8414" data-size="0" data-startline="55">Label catalog number (string)</li> <li data-endline="56" data-position="8446" data-size="0" data-startline="56">Album title</li> <li data-endline="57" data-position="8460" data-size="0" data-startline="57">Track title</li> <li data-endline="58" data-position="8474" data-size="0" data-startline="58">Track number</li> <li data-endline="59" data-position="8489" data-size="0" data-startline="59">Unsynced lyrics</li> <li data-endline="60" data-position="8507" data-size="0" data-startline="60">Key (dropdown)</li> <li data-endline="61" data-position="8524" data-size="0" data-startline="61">BPM (currently a string, should be an integer)</li> <li data-endline="62" data-position="8573" data-size="0" data-startline="62">License (dropdown, missing CC0)</li> <li data-endline="63" data-position="8607" data-size="0" data-startline="63">ISRCs</li> <li data-endline="64" data-position="8615" data-size="0" data-startline="64">Genres (global folksonomy of genres! cool!)</li> <li data-endline="66" data-position="8661" data-size="0" data-startline="65">External links</li> </ul>
<p>Regarding data collection, this is already an improvement over other digital storefronts, but has some big implementation pitfalls:</p>
<ol data-endline="77" data-position="8810" data-size="0" data-startline="69"> <li data-endline="71" data-position="8813" data-size="0" data-startline="69">Metadata tagging is fractured across two steps of the upload flow <ul data-position="8884" data-size="0"> <li data-endline="70" data-position="8886" data-size="0" data-startline="70">Instrument / studio musician credits are often per-track <em data-position="8943" data-size="0">or</em> just listed for the full album. These are currently inherited by each track in a separate track editing flow and could be combined into a single credit with the option to specify track numbers instead of entering this information on two seperate screens.</li> <li data-endline="71" data-position="9209" data-size="0" data-startline="71">This appears to have been implemented due to the seperate upload flows for singles and albums. These should be reconsidered and all options should be incorporated into a single upload flow for creating a "release" rather than a "single" or an "album". Let users choose the release type (as they do currently) or determine it for them if they only upload one track.</li> </ul> </li> <li data-endline="73" data-position="9577" data-size="0" data-startline="72">"Label" values are stored only with human readable names. <ul data-position="9639" data-size="0"> <li data-endline="73" data-position="9641" data-size="0" data-startline="73">Given that labels are an important part of Subvert's future platform features, ensuring that duplication is discouraged is important. We want to be able to authoritatively link to labels!</li> </ul> </li> <li data-endline="77" data-position="9832" data-size="0" data-startline="74">Credit metadata is not structured and therefore cannot be written to files according to tagging specs expected by music players. <ul data-position="9965" data-size="0"> <li data-endline="77" data-position="9967" data-size="0" data-startline="75">The ID3 tagging spec commonly used for music is <em data-position="10015" data-size="0">quite rigid</em>. Subvert needs to know what the tag represents to assign it to the correct tag that music players will expect to see. <ul data-position="10155" data-size="0"> <li data-endline="77" data-position="10157" data-size="0" data-startline="76">Artists will spend a good deal of time entering this, it should make its way into the end product!</li> </ul> </li> </ul> </li> </ol>
<h2 data-endline="78" data-id="Recommendations" data-startline="78">Recommendations</h2>
<p>I believe Subvert's primary metadata-related goals should be as follows:</p>
<ol data-endline="85" data-position="10351" data-size="0" data-startline="82"> <li data-endline="82" data-position="10354" data-size="0" data-startline="82">Provide customers with industry leading metadata for downloaded tracks</li> <li data-endline="83" data-position="10428" data-size="0" data-startline="83">Empower artists to accurately credit collaborators</li> <li data-endline="85" data-position="10482" data-size="0" data-startline="84">Enable customers to search for and find the music they love</li> </ol>
<p>In my view, the purchase flow for listeners and the upload flow for artists are the key launch user experiences that will be integral to Subvert's success. Metadata entry is of course part of the latter; making the form easy to use for artists will translate into an improved end product for fans: A properly tagged, high quality music file.</p>
<p>To enable Subvert, both at launch and in the future, to follow through on these desires, I recommend the following changes:</p>
<p>Different file formats have different tagging formats. To deal with this unfortunate reality, MusicBrainz uses internal names for their tagging system and <a href="https://picard-docs.musicbrainz.org/en/appendices/tag_mapping.html">maps them to each different tagging spec</a>. This list also uses MusicBrainz' internal names and I would recommend you adopt the same spec. 20+ years of thinking has been put into this already.</p>
<p>"Mandatory" tags are ones that everyone would rightfully expect to be added or are otherwise "free" requiring no additional input from the user. Some of them may be editable in the form.</p>
<p>"Optional" tags should be available in the upload form as non-mandatory fields.</p>
<p>These tags should be written to every file.</p>

<p>These tags should be written on a per-track basis if entered by the artist. While MusicBrainz supports <em data-position="15759" data-size="0">many</em> other artist relationships, these are the only ones that make it into file metadata and therefore the most important ones for launch.</p>

<p>Subvert should implement MusicBrainz' album credit relationships spec for keys, but leave artist name values open and not tied to unique identifiers unless a Subvert user ID is optionally linked. This scope ensures that artists on the platform can be linked to properly from credit fields and that their contributions can be organized for their profile pages but it does not prevent name-collisions when searching. If off-platform credited artists were to be unique (like MusicBrainz) you would need to implement extensive merging, voting, and history tools to allow the community to keep the database accurate should a credit be assigned to the wrong artist or if multiple artist entries are created in error. Limiting unique linked-data to on-platform linked credits keeps management of credits to the album owner and (ideally) the linked artist.</p>
<p>This data structure for credits could be implemented as follows:</p>
<p>Instrument Type should pull from the MusicBrainz instrument tree available in <a href="https://data.metabrainz.org/pub/musicbrainz/data/json-dumps/">their data dumps directory</a>. This should allow artists a <em data-position="19044" data-size="0">lot</em> of flexibility! Relying on it as a dependency will allow you to adopt its extensive and localized data structure without the serious maintnance burdeon, expect this to be updated over time.</p>
<p>Also of note is the less comprehensive <a href="https://service.ddex.net/dd/DD-AVS-CURRENT/dd/avs_InstrumentType.html">DDEX instrument list</a> for future integration with DDEX standards. I'm unaware of efforts to map values from MusicBrainz' list to DDEX's, though the DDEX standard allows for arbitrary entries.</p>
<p>Unlike artist relationship credits, I recommend that labels DO get assigned a unique identifier. There are significantly fewer labels than artists — I think name collision issues are unlikely — and this will allow labels to be assigned this value in the future to map and link their name to their label account. Encourage artists to assign their tracks to existing labels to make the process of adopting them later easier!</p>

<h2 data-endline="289" data-id="Future-Opportunities" data-startline="289">Future Opportunities</h2>
<p>Perhaps the most obvious on-platform opportunity! Giving artists the ability to link to other credited artists that are on the platform is great for letting fans find more work from artists they connect with. The first step toward this should be linking on-platform artist credits in the sidebar.</p>
<p>Beyond linking, cleanly organized structured data is the foundational step toward building a recommendation algorithm. The more accurately tagged dimensions one has to work with, the more specific the recommendation algorithm can be! Given a vast quantity of data, the better the recommendations will become.</p>
<p>Using <a href="https://www.npmjs.com/package/node-id3">Node ID3</a> you can read and auto-populate as many areas of the upload form as possible if artists upload pre-tagged files! What a time saver!</p>
<p>Linking credits to both names and optionally Subvert user CUIDs could be the start of the royalty split distribution flow. I imagine after entering metadata, on a different page users could be presented with all of the contributors on the album and could choose how to allocate revenue as percentages for each linked artist.</p>
<p>We can't always implement <em data-position="22818" data-size="0">everything</em> in one go! Here's a few fields that I thin you can reasonably leave for the future:</p>

<p>The <a href="https://kb.ddex.net/implementing-each-standard/">DDEX metadata standards</a> are a more modern (and more comprenehsive) group of definitions for music metadata spanning artists, releases, labels, and more. When implementing and expanding upon these features in the future, I would recommend consulting their data structures to help enable future interoperability. The <a href="https://service.ddex.net/dd/DD-AVS-CURRENT/dd/intro.html">allowed value set dictionary</a> is handy to make sure your related data structures will be spec compliant.</p>
