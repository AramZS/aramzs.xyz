title: Trusted publishing for npm packages

<p>Trusted publishing allows you to publish npm packages directly from your CI/CD workflows using <a data-component="Link" href="https://openid.net/developers/how-connect-works/">OpenID Connect (OIDC)</a> authentication, eliminating the need for long-lived npm tokens. This feature implements the <a data-component="Link" href="https://repos.openssf.org/trusted-publishers-for-all-package-repositories">trusted publishers industry standard</a> specified by the Open Source Security Foundation (OpenSSF), joining a growing ecosystem including <a data-component="Link" href="https://docs.pypi.org/trusted-publishers/">PyPI</a>, <a data-component="Link" href="https://guides.rubygems.org/trusted-publishing/">RubyGems</a>, and other major package registries in offering this security enhancement.</p>
<p><strong>Note:</strong> Trusted publishing requires <a data-component="Link" href="https://docs.npmjs.com/cli/v11">npm CLI</a> version 11.5.1 or later and Node version 22.14.0 or higher.</p>
<p>Trusted publishing creates a trust relationship between npm and your CI/CD provider using OIDC. When you configure a trusted publisher for your package, npm will accept publishes from the specific workflow you've authorized, in addition to traditional authentication methods like npm tokens and manual publishes. The npm CLI automatically detects OIDC environments and uses them for authentication before falling back to traditional tokens.</p>
<p>This approach eliminates the security risks associated with long-lived write tokens, which can be compromised, accidentally exposed in logs, or require manual rotation. Instead, each publish uses short-lived, cryptographically-signed tokens that are specific to your workflow and cannot be extracted or reused.</p>

<p>You can modify or remove your trusted publisher configuration at any time through your package settings on <a data-component="Link" href="https://npmjs.com">npmjs.com</a> → Packages → YOUR_PACKAGE → Settings → Trusted publishing. Each package can only have one trusted publisher connection at a time, but this connection can be edited or deleted as needed. To change providers (for example, switching from GitHub Actions to GitLab CI/CD), simply edit your existing configuration and select the new provider. The change takes effect immediately for future publishes. To remove trusted publishing entirely and return to token-based authentication, delete the trusted publisher configuration from your package settings.</p>
<p>Once you've configured trusted publishers for your package, we strongly recommend restricting traditional token-based publishing access for enhanced security.</p>
<ol> <li>After enabling trusted publishers, navigate to your package's <strong>Settings</strong> → <strong>Publishing access</strong></li> <li>Select <strong>"Require two-factor authentication and disallow tokens"</strong></li> <li>Save your changes by clicking on <strong>Update Package Settings</strong></li> </ol>
<p>Trusted publishers use short-lived, scoped credentials that are generated on-demand during your CI/CD workflow, eliminating the need for long-lived tokens. By restricting traditional token access while using trusted publishers, you reduce potential security risks associated with credential management.</p>
<p>For even stronger security, configure your trusted publisher with <strong>stage-only permissions</strong> (allow <code>npm stage publish</code> but not <code>npm publish</code>). This ensures all CI-automated publishes go through the <a data-component="Link" href="https://docs.npmjs.com/staged-publishing">staged publishing</a> flow, requiring a maintainer to review and approve each package with 2FA via the CLI or <a data-component="Link" href="https://www.npmjs.com">npmjs.com</a> before it becomes publicly available. Combined with disallowing tokens, this provides the maximum security posture for your packages.</p>
<p><strong>Note:</strong> The "disallow tokens" setting only affects traditional token authentication. Your trusted publishers will continue to work normally, as they use OIDC tokens.</p>
<p>If you're transitioning from token-based publishing:</p>
<ol> <li>Set up trusted publishers first and verify they work</li> <li>Then restrict token access as described above</li> </ol>
<p>This ensures a smooth transition without disrupting your release process.</p>
<p>When you publish using trusted publishing from GitHub Actions or GitLab CI/CD, npm automatically generates and publishes <a data-component="Link" href="https://docs.npmjs.com/generating-provenance-statements">provenance attestations</a> for your package. This happens by default—you don't need to add the <code>--provenance</code> flag to your publish command.</p>
<p>Provenance provides cryptographic proof of where and how your package was built, allowing users to verify its authenticity. This automatic generation only applies when all of these conditions are met:</p>
<ul> <li>Publishing via trusted publishing (OIDC)</li> <li>Publishing from a public repository</li> <li>Publishing a public package</li> </ul>

<p>When trusted publishing is available for your workflow, always prefer it over long-lived tokens. Traditional npm tokens pose several security risks:</p>
<ul> <li>They can be accidentally exposed in CI logs or configuration files</li> <li>They require manual rotation and management</li> <li>If compromised, they provide persistent access until revoked</li> <li>They often have broader permissions than necessary</li> </ul>
<p>Trusted publishing eliminates these risks by using short-lived, workflow-specific credentials that are automatically managed and cannot be extracted.</p>
<p>While trusted publishing handles the publish operation, you may still need authentication for installing private npm dependencies. For this scenario, we recommend:</p>
<p>Always use <a data-component="Link" href="https://docs.npmjs.com/creating-and-viewing-access-tokens#creating-granular-access-tokens-on-the-website">read-only granular access tokens</a> for installing dependencies. This limits potential damage if the token is ever compromised.</p>
<p>Consider implementing these additional security practices:</p>
<ul> <li>Enable <a data-component="Link" href="https://docs.github.com/en/repositories/managing-your-repositorys-settings-and-features/managing-repository-settings/configuring-tag-protection-rules">tag protection rules</a> to control who can create release tags</li> <li>Regularly audit your trusted publisher configurations</li> <li>Remove any unused publish tokens from your npm account</li> </ul>
<p>If you encounter an "Unable to authenticate" (ENEEDAUTH) error when publishing, first verify that the workflow filename matches exactly what you configured on <a data-component="Link" href="https://npmjs.com">npmjs.com</a>, including the <code>.yml</code> extension. All fields are case-sensitive and must be exact. Also ensure you're using GitHub-hosted runners, GitLab.com shared runners, or CircleCI cloud, as self-hosted runners are not currently supported. For GitHub Actions specifically, check that the <code>id-token: write</code> permission is set in your workflow. For CircleCI, verify that your Organization ID, Project ID, and Pipeline definition ID match your CircleCI configuration.</p>
<p>To publish from GitHub, your package's <code>repository.url</code> field in <code>package.json</code> must exactly match your GitHub repository. This may be an issue for misconfigured packages, but could also impact publication from forks that haven't updated <code>package.json</code> to match the forked repo.</p>
<p>If your package has private dependencies and <code>npm install</code> or <code>npm ci</code> is failing with authentication errors, remember that trusted publishing only applies to the <code>npm publish</code> command. You'll still need to provide a read-only token for installing private packages as shown in the examples above.</p>
<p>For packages in private repositories, provenance will not be generated even though you're using trusted publishing. This is a <a data-component="Link" href="https://github.blog/changelog/2023-07-25-publishing-with-npm-provenance-from-private-source-repositories-is-no-longer-supported/">known limitation</a> that applies regardless of whether your package itself is public or private.</p>
<p>Some GitHub Actions workflows use <code>workflow_call</code> to invoke other workflows that run <code>npm publish</code>, or use <code>workflow_dispatch</code> for manual publishing. When this happens, validation checks the calling workflow's name instead of the workflow that actually contains the publish command, which can cause configuration mismatches. The <code>id-token: write</code> permission must also be given to both parent and child workflows.</p>
<p>Trusted publishing currently supports only cloud-hosted runners. Support for self-hosted runners is intended for a future release. Each package can only have one trusted publisher configured at a time, though you can update this configuration as needed.</p>
<p>OIDC authentication supports the <code>npm publish</code> and <code>npm stage publish</code> commands. Other stage subcommands (<code>npm stage list</code>, <code>npm stage view</code>, <code>npm stage approve</code>, <code>npm stage reject</code>) require interactive authentication and cannot use OIDC tokens, as these actions require proof of presence and can only be performed via the CLI or <a data-component="Link" href="https://www.npmjs.com">npmjs.com</a>. Other npm commands such as <code>install</code>, <code>view</code>, or <code>access</code> still require traditional authentication methods. The <code>npm whoami</code> command will not reflect OIDC authentication status since the authentication occurs only during the publish or stage operation.</p>
<p>We intend to expand trusted publishing support to additional CI/CD providers and enhance the feature based on community feedback.</p>
