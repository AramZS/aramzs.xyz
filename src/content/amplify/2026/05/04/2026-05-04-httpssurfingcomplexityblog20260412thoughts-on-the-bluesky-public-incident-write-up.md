<p>Back on April 4, the social media site Bluesky suffered a pretty big outage. I was delighted to discover that one of their engineers, <a href="https://bsky.app/profile/did:plc:4uz2445cjiw7w4nobfgnu35f">Jim Calabro</a>, published a public writeup about it: <a href="https://pckt.blog/b/jcalabro/april-2026-outage-post-mortem-219ebg2">April 2026 Outage Post-Mortem</a>.</p>
<p>Calabro’s post goes into a lot of technical details about the failure mode. I’m using this post as a learning exercise for myself. I find that if I have to explain something, then I’ll understand it better. After reading his post and writing this one, I learned things about ephemeral ports, goroutine groups, the TCP state machine, the interaction between blocking system calls and the creation of threads in the Go runtime, and the range of loopback addresses on Linux.</p>
<h2>Interpreting the error message </h2>
<p>The first thing that struck me is Calabro’s write-up was his discussion of a particular error message he saw in the logs:</p>
<pre><code>dial tcp 127.32.0.1:0-&gt;127.0.0.1:11211: bind: address already in use</code></pre>
<p>Now, if I was the one who saw the error message “bind: address already in use”, I would have assumed that a process was trying to listen on a port that another process was already listening on. This sort of thing is server-side behavior, where a server listens on a port (e.g., web servers listen on port 80 and port 443). In the connect attempt associated with the log, the server is listening on port 11211 (the standard port used by <a href="https://www.memcached.org/">memcached</a>). As it says on the Linux <a href="https://www.man7.org/linux/man-pages/man2/bind.2.html"><em>bind</em> man page</a>:</p>
<pre><strong> EADDRINUSE<br/></strong>              The given address is already in use.</pre>
<p>But that wasn’t the problem in this case! It wasn’t an issue with a server trying and failing to listen on port 11211. Instead, the problem is that the <strong><em>client</em></strong>, which is trying to make a connection to the memcached service, is failing to associate a socket with a port. The system call that’s failing is not <em>listen</em> but (as indicated in the error message) <em>bind</em>. That bind man page actually has two different entries for the <em>address already in use</em> error. Here’s the second one:</p>
<pre><strong>EADDRINUSE</strong><br/>              (Internet domain sockets) The port number was specified as<br/>              zero in the socket address structure, but, upon attempting<br/>              to bind to an ephemeral port, it was determined that all<br/>              port numbers in the ephemeral port range are currently in<br/>              use. See the discussion of<br/>              /proc/sys/net/ipv4/ ip_local_port_range <a href="https://www.man7.org/linux/man-pages/man7/ip.7.html">ip(7)</a>.</pre>
<p>I assume that go’s net.Dial function ultimately calls this <a href="https://github.com/golang/go/blob/master/src/net/sock_posix.go?utm_source=chatgpt.com#L89">private dial function</a>, which will call <em>bind</em> if the caller explicitly specifies the local address. In the log message above, the local address was <code>127.32.0.1:0</code>.</p>
<p>This code was failing because there were no available ephemeral ports left!</p>
<p>I bring this up because Calabro simply mentions as an aside how he (correctly!) interpreted the error message. He just shows the error, and then writes (emphasis mine):</p>
<blockquote> <p>The timing of these log spikes lined up with drops in user-facing traffic, which makes sense. Our data plane <em>heavily</em> uses memcached to keep load off our main Scylla database, and if we’re <strong>exhausting ports</strong>, that’s a huge problem.</p> </blockquote>
<p>That’s expertise in action!</p>
<h2>Saturation, part 1: ephemeral ports</h2>
<p>The failure mode that Bluesky encountered is a classic example of <a href="https://resilienceinsoftware.org/news/11475336">saturation</a>, where the system runs out of a critical resource. Calabro’s write-up covers two different time periods, a paging alert on Saturday April 4, and then the Bluesky outage that happened two days later, on Monday April 6. There were different flavors of saturation on the different days, here we’ll talk about the first one.</p>
<p>On Saturday, the limited resource in question was the number of available ephemeral ports. From a programming perspective, when we make calls to servers, we don’t think about the fact that our side of a TCP connection gets assigned a port, because this TCP detail is effectively abstracted away from the developer.</p>
<p>I’m running on macOS, but if I launch an Ubuntu Docker container, I can see that the ephemeral port range goes from 32768 to 60999, for a count of 28,232 available ephemeral ports:</p>
<p>The irony here is that the connections that exhausted the ephemeral ports were to a process that’s running on the same host: memcached listening on 127.0.0.1:11211.</p>
<p>Calabro goes into considerable detail about how the service they refer to as the <em>data plane</em> ran out of ephemeral ports. I’ll describe my understanding based on his write-up. But, as always, I recommend you read the original.</p>
<p>The data plane service talks to a database that is fronted by memcached. This incident only involved interactions between data plane and memcached, so I don’t show the database in the diagram below.</p>
<figure><a href="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image.png"><img alt="" data-attachment-id="7493" data-comments-opened="1" data-image-caption="" data-image-description="" data-image-meta='{"aperture":"0","credit":"","camera":"","caption":"","created_timestamp":"0","copyright":"","focal_length":"0","iso":"0","shutter_speed":"0","title":"","orientation":"0","alt":""}' data-image-title="image" data-large-file="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image.png?w=1024" data-orig-file="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image.png" data-orig-size="1203,630" data-permalink="https://surfingcomplexity.blog/2026/04/12/thoughts-on-the-bluesky-public-incident-write-up/image-510/" sizes="(max-width: 1024px) 100vw, 1024px" src="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image.png" srcset="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image.png?w=1024 1024w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image.png?w=150 150w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image.png?w=300 300w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image.png?w=768 768w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image.png 1203w"/></a><figcaption>How the data plane service ran out of ephemeral ports</figcaption></figure>
<p>Bluesky recently brought up a new internal service. One of the things this service does is make the GetPostRecord RPC call against the data plane service. The problem isn’t with the rate of traffic. In fact, the volume of traffic that this internal service sends to data plane is low, less than 3 RPS.</p>
<p>No, the problem here is the size of the GetPostRecord payload. It sends a batch of URIs in each call, and sometimes those batches are very large, on the order of 15-20 thousand URIs.</p>
<p>The data plane looks up each URI in memcache first before hitting the database. The data plane is written in Go, and for each request, it starts a new goroutine, and each of those goroutines creates a new TCP connection to memcache. All of those goroutines concurrently making those TCP connections depleted the set of available ephemeral ports.</p>
<p>One thing I learned from this write-up is that Go has a notion of <a href="https://pkg.go.dev/golang.org/x/sync/errgroup#Group">goroutine groups</a>, you can explicitly set a limit of the number of goroutines that are active within a given group. Tragically, this was the one data plane endpoint that was missing an explicit limit.</p>
<p>In the write-up, Calabro notes that the memcached client uses a connection pool, with a maximum idle size of 1000 connections. I was initially confused by this, because I’m used to connection pools where the pool defines the maximum number of simultaneous active connections, and if no unused connections are available, then the client blocks waiting for a connection to be available.</p>
<p>I looked into this, and assuming that this app is using the <a href="https://github.com/bradfitz/gomemcache/tree/master">gomemcache</a> library, that’s not how its connection pool works. Instead, the <a href="https://github.com/bradfitz/gomemcache/blob/master/memcache/memcache.go#L294-L299">gomemcache code</a> first looks to see if there’s an available connection. If not, it creates a new connection. So, the connection pool here doesn’t bound connections, but rather is an optimization to reuse an existing connection if one is available.</p>
<p>Instead, what you specify with gomemcache is the <a href="https://github.com/bradfitz/gomemcache/blob/8d39553ac7cf/memcache/memcache.go#L148-L154">maximum number of idle connections</a>, which is the maximum number of connections that the pool will hold onto after use. As mentioned above, Bluesky had this configured as 1,000. This means that if there are 15,000 new connections requested concurrently, at best 1,000 connections will be reused from the pool, requiring 14,000 new connections to be established.</p>
<h2>Bitten by time lags – TIME_WAIT</h2>
<p>Time lags are underrated factor in incidents, and time lag plays a role here. In this case, the time lag is due to a state in the lifetime of a TCP socket called <a href="https://serverframework.com/asynchronousevents/2011/01/time-wait-and-its-design-implications-for-protocols-and-scalable-servers.html">TIME_WAIT</a>. This state renders a port unusable for a fixed period of time after a connection associated with the port has been closed.</p>
<p>Personally, I first encountered TIME_WAIT back when I was working on a web app on my laptop. Sometimes I’d kill the process and restart it, and the restart would fail with the error that the port it was trying to listen on was already in use. It turns out that the operating system does not immediately release the ports associated with a socket after it’s closed. Instead, the connection transitions to the TIME_WAIT state.</p>
<p>Here’s an explanation for why TIME_WAIT exists, based largely on the excellent article: <a href="https://serverframework.com/asynchronousevents/2011/01/time-wait-and-its-design-implications-for-protocols-and-scalable-servers.html">TIME_WAIT and its design implications for protocols and scalable client server systems</a> from <a href="https://serverframework.com/">ServerFramework.com</a>.</p>
<h3>The dropped ACK problem: sending an error when nothing is wrong</h3>
<p>Closing a TCP requires each send side to send a FIN, and each side to ACK the received FIN. As each side sends or receives one of these packets, it transitions through the TCP state machine. Here’s what the exchange looks like. I’ve annotated the TCP states on the server side and the client side.</p>
<figure><a href="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-15.png"><img alt="" data-attachment-id="7548" data-comments-opened="1" data-image-caption="" data-image-description="" data-image-meta='{"aperture":"0","credit":"","camera":"","caption":"","created_timestamp":"0","copyright":"","focal_length":"0","iso":"0","shutter_speed":"0","title":"","orientation":"0","alt":""}' data-image-title="image" data-large-file="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-15.png?w=1024" data-orig-file="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-15.png" data-orig-size="1060,490" data-permalink="https://surfingcomplexity.blog/2026/04/12/thoughts-on-the-bluesky-public-incident-write-up/image-525/" sizes="(max-width: 1024px) 100vw, 1024px" src="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-15.png" srcset="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-15.png?w=1024 1024w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-15.png?w=150 150w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-15.png?w=300 300w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-15.png?w=768 768w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-15.png 1060w"/></a><figcaption>What state should the client be in after receiving the FIN?</figcaption></figure>
<p>It looks like the client should also be in the CLOSED state after it receives the FIN. However, that creates a problem if the ACK it sends never makes it, because the server will eventually retry sending the FIN.</p>
<figure><a href="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-16.png"><img alt="" data-attachment-id="7549" data-comments-opened="1" data-image-caption="" data-image-description="" data-image-meta='{"aperture":"0","credit":"","camera":"","caption":"","created_timestamp":"0","copyright":"","focal_length":"0","iso":"0","shutter_speed":"0","title":"","orientation":"0","alt":""}' data-image-title="image" data-large-file="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-16.png?w=1024" data-orig-file="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-16.png" data-orig-size="1103,750" data-permalink="https://surfingcomplexity.blog/2026/04/12/thoughts-on-the-bluesky-public-incident-write-up/image-526/" sizes="(max-width: 1024px) 100vw, 1024px" src="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-16.png" srcset="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-16.png?w=1024 1024w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-16.png?w=150 150w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-16.png?w=300 300w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-16.png?w=768 768w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-16.png 1103w"/></a></figure>
<p>Here the client has received a packet associated with a TCP connection that has transitioned to the CLOSED state. The client will treat this as an error, and will send an RST packet (if you’ve ever seen the message: <em>connection reset by peer</em>, you’ve been on the receiving end of an RST packet).</p>
<p>To prevent this, after sending an ACK in the FIN_WAIT_2, the client transitions into the TIME_WAIT state. From <a href="https://www.ietf.org/rfc/rfc9293.html#FTY99">RFC-9293</a>:</p>
<blockquote> <p>When a connection is closed actively, it MUST linger in the TIME-WAIT state for a time 2xMSL (Maximum Segment Lifetime)</p> </blockquote>
<p>The RFC doesn’t define what the maximum segment lifetime is. On Linux, the kernel waits in the TIME_WAIT state for about <a href="https://github.com/torvalds/linux/blob/master/include/net/tcp.h#L142-L143">60 seconds</a>.</p>
<p>This means that the state of the TCP connection will be in the TIME_WAIT state for about a minute before transitioning to CLOSED:</p>
<figure><a href="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-17.png"><img alt="" data-attachment-id="7556" data-comments-opened="1" data-image-caption="" data-image-description="" data-image-meta='{"aperture":"0","credit":"","camera":"","caption":"","created_timestamp":"0","copyright":"","focal_length":"0","iso":"0","shutter_speed":"0","title":"","orientation":"0","alt":""}' data-image-title="image" data-large-file="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-17.png?w=1024" data-orig-file="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-17.png" data-orig-size="1401,741" data-permalink="https://surfingcomplexity.blog/2026/04/12/thoughts-on-the-bluesky-public-incident-write-up/image-527/" sizes="(max-width: 1024px) 100vw, 1024px" src="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-17.png" srcset="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-17.png?w=1024 1024w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-17.png?w=150 150w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-17.png?w=300 300w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-17.png?w=768 768w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-17.png 1401w"/></a></figure>
<h3>The out of order problem: packet associated with wrong connection</h3>
<p>TIME_WAIT also deals with a problem related to packets being received out of order.</p>
<p>Note that a TCP connection’s identity is determined by the four-tuple: (source IP, source port, destination IP, destination port). Here’s an example of such a four-tuple: <code>(127.32.0.1, 32768, 127.0.0.1, 11211)</code>.</p>
<p>Because TCP packets can arrive out of order, there might still be packets in-flight associated with that connection. If a new TCP connection with the same four-tuple is opened, the receiver will incorrectly associate the packet with the new connection, even though it was part of the old one, as depicted below (here I’m simplifying the connect and close to a single packet rather than using three packets).</p>
<figure><a href="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-1.png"><img alt="" data-attachment-id="7501" data-comments-opened="1" data-image-caption="" data-image-description="" data-image-meta='{"aperture":"0","credit":"","camera":"","caption":"","created_timestamp":"0","copyright":"","focal_length":"0","iso":"0","shutter_speed":"0","title":"","orientation":"0","alt":""}' data-image-title="image" data-large-file="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-1.png?w=1024" data-orig-file="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-1.png" data-orig-size="1061,440" data-permalink="https://surfingcomplexity.blog/2026/04/12/thoughts-on-the-bluesky-public-incident-write-up/image-511/" sizes="(max-width: 1024px) 100vw, 1024px" src="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-1.png" srcset="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-1.png?w=1024 1024w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-1.png?w=150 150w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-1.png?w=300 300w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-1.png?w=768 768w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-1.png 1061w"/></a><figcaption>The blue “send” packet is incorrectly associated with the green TCP connection.</figcaption></figure>
<p>TIME_WAIT also prevents this by having the client enter TIME_WAIT that is long enough to guarantee that the sent packet is received before the new connection can be opened on the same port.</p>
<h3>Eating up the ephemeral port space</h3>
<p>Because you have to wait about a minute before you can reuse an ephemeral port, TIME_WAIT reduces the amount of available ephemeral ports.</p>
<p>Returning to the Bluesky scenario, imagine that the memcached connection pool is fully populated (there are 1000 idle connections ready to be used), and the rest of the ephemeral ports are free. I’ll depict the space of 28,232 ephemeral ports as a rectangle, with the green rectangle indicating the connection pool.</p>

<p>The maximum idle connections is set to 1000, so 1000 of the active connections get returned to the pool. The rest of the connections are closed, and eventually enter the TIME_WAIT state:</p>

<p>Now, another wave of connection requests comes in. Because the ephemeral ports are in use by TCP connections in the TIME_WAIT state, they’re unavailable:</p>

<p>You can see how the ephemeral ports could be consumed if large numbers of connection requests came in one after another before the TIME_WAIT timer elapsed.</p>
<h2>Saturation, part 2: memory</h2>
<p>While Bluesky observed the problem with ephemeral port exhaustion on Saturday, it wasn’t until the Monday that they suffered from an outage.</p>
<p>From the write-up, it’s not clear to me what exactly changed on Monday. Perhaps it was just an organic increase in traffic that exacerbated the problem? Whatever it was, the ephemeral port exhaustion contributed to a cascading failure.</p>
<figure><a href="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-12.png"><img alt="" data-attachment-id="7531" data-comments-opened="1" data-image-caption="" data-image-description="" data-image-meta='{"aperture":"0","credit":"","camera":"","caption":"","created_timestamp":"0","copyright":"","focal_length":"0","iso":"0","shutter_speed":"0","title":"","orientation":"0","alt":""}' data-image-title="image" data-large-file="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-12.png?w=1024" data-orig-file="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-12.png" data-orig-size="1954,167" data-permalink="https://surfingcomplexity.blog/2026/04/12/thoughts-on-the-bluesky-public-incident-write-up/image-522/" sizes="(max-width: 1024px) 100vw, 1024px" src="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-12.png" srcset="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-12.png?w=1024 1024w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-12.png?w=150 150w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-12.png?w=300 300w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-12.png?w=768 768w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-12.png?w=1440 1440w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-12.png 1954w"/></a><figcaption>According to the write-up, the failure cascade went something like this:</figcaption></figure>
<ol> <li>The ephemeral port exhaustion led to error messages when attempting to call memcached.</li> <li>Every memcached error resulted in a log line being written synchronously to disk.</li> <li>A large number of goroutines blocked in synchronous system calls led to the Go runtime spawning many OS-level threads (I learned that OS-level threads are <a href="https://go.dev/src/runtime/HACKING">called <em>M</em> in Go parlance</a>).</li> <li>This large number of OS-level threads put memory pressure on the app.</li> <li>As a result, the data plane experienced stop-the-world GC pauses as well as OOM kills.</li> </ol>
<p>Note that because TIME_WAIT is an OS-level state, a data plane process that was OOM killed and restarted would still face limits on the ephemeral port space!</p>
<h2>The workaround: leveraging multiple loopbacks</h2>
<p>I was impressed by their improvised solution to deal with the problem. I’ve been talking about how an ephemeral port can be consumed, but it’s not actually the port itself. When calling the <em>bind</em> function, you provide not just a port, but the local IP address you want to bind to. It’s the (IP, port) pair that is limited, not the port.</p>
<p>So, if you want to create a TCP connection to a local process (like, say, memcache), and the pair (127.0.0.1,32768) is already in use, if there are other IP addresses that are <em>loopback</em> addresses, you can use those too!</p>
<p>On Linux, by default, all 128.*.*.* IP addresses are loopback address!</p>
<pre><code>
# ip route show table local
local 127.0.0.0/8 dev lo proto kernel scope host src 127.0.0.1
...</code></pre>
<p>(Note that this is different from macOS, which only routes 127.0.0.1 via loopback by default).</p>
<p>This means that you potentially have access to a much larger space of ephemeral ports!</p>
<figure><a href="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-14.png"><img alt="" data-attachment-id="7539" data-comments-opened="1" data-image-caption="" data-image-description="" data-image-meta='{"aperture":"0","credit":"","camera":"","caption":"","created_timestamp":"0","copyright":"","focal_length":"0","iso":"0","shutter_speed":"0","title":"","orientation":"0","alt":""}' data-image-title="image" data-large-file="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-14.png?w=1024" data-orig-file="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-14.png" data-orig-size="1096,409" data-permalink="https://surfingcomplexity.blog/2026/04/12/thoughts-on-the-bluesky-public-incident-write-up/image-524/" sizes="(max-width: 1024px) 100vw, 1024px" src="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-14.png" srcset="https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-14.png?w=1024 1024w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-14.png?w=150 150w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-14.png?w=300 300w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-14.png?w=768 768w, https://surfingcomplexity.blog/wp-content/uploads/2026/04/image-14.png 1096w"/></a></figure>
<p>Applying terminology from resilience engineering, ephemeral ports are a resource, and you have to do work to <em>marshal</em> these additional resources.</p>
<p>For Bluesky, the work of marshaling resources came in the form of modifying the code that made the TCP connections. They modified it to randomly select a loopback IP address. Here’s the code from the blog post:</p>
<pre><code>memcachedClient.DialContext = func(ctx context.Context, network, address string) (net.Conn, error) {<br/>	ip := net.IPv4(127, byte(1+rand.IntN(254)), byte(rand.IntN(256)), byte(1+rand.IntN(254)))<br/>	d := net.Dialer{LocalAddr: &amp;net.TCPAddr{IP: ip}}<br/>	return d.DialContext(ctx, network, address)</code></pre>
<p>Calabro’s describes the above change as:</p>
<blockquote> <p>The band-aid fix was insane but did the job. </p> </blockquote>
<p>I wouldn’t describe this is insane, though. This is exactly the kind of improvisational work that you frequently have to do in order to get a system back to healthy during the incident.</p>
<p>Calabro briefly discusses how difficult it was to diagnose the issue, emphasis mine:</p>
<blockquote> <p>It was all buried in there, but it was hard to know where to look when <strong>so much was falling over all at once</strong>. You need to have the mental discipline and high granularity in your metrics to be able to cut through the noise to find the real root cause. It’s hard work!</p> </blockquote>
<p>I wish there had been more in this writeup about the process the engineers went through to actually figure out what was going on during the incident, because descriptions of diagnostic work is one of my favorite parts of incident write-ups. We all can stand to do better at improving our diagnostic skills, and one way I try to improve is to read about how someone diagnosed an issue during an incident.</p>
<p>As Calabro mentions, during an incident, there are frequently many things that are failing, and it can be extremely hard to tease out the signals that will help you understand how the system first got into this state.</p>
<p>One particular challenge is noticing an error signal that happens to be unrelated to the ongoing incident, as happened during this incident (emphasis mine):</p>
<blockquote> <p>EDIT: Also, the status page said this was an issue with a 3rd party provider. It was clearly not, apologies for that miscommunication! At the time I posted that status page update, I was looking at some traceroutes that indicated some <strong>pretty substantial packet loss from a cloud provider to our data center</strong>, but those <strong>were not the root cause of the issue</strong>.</p> </blockquote>
<h2>The messy 9</h2>
<p>I want to end this post by bringing up the <a href="https://www.thisisfinepod.com/the-pod/the-messy-9-and-coding-with-ai-a-panel-discussion">Messy 9</a>, a set of patterns proposed by the resilience engineering researcher David Woods. These are:</p>
<ol> <li>congestion</li> <li>cascades</li> <li>conflicts</li> <li>saturation</li> <li>lag</li> <li>friction</li> <li>tempos</li> <li>surprises</li> <li>tangles</li> </ol>
<p>I’ve explicitly discussed <em>cascades, saturation,</em> and <em>lag</em> in this post. I suspect that, if we had more detail about this incident, we’d identify even more of these patterns here. Keep on the look-out for these the next time you read an incident write-up or attend an incident review meeting!</p>
